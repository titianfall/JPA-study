## 1. JPA 구동 방식 (제일 중요)

```
Persistence  ──(1. 설정 조회)──▶  META-INF/persistence.xml
     │
     │ (2. 생성)
     ▼
EntityManagerFactory (EMF)  ──(3. 생성)──▶  EntityManager (EM) × N
```

- **`Persistence`** : `persistence.xml` 설정을 읽어서 `EntityManagerFactory`를 만드는 시작점
- **`EntityManagerFactory` (EMF)** : `EntityManager`를 찍어내는 공장
- **`EntityManager` (EM)** : 실제 DB 작업(등록/수정/삭제/조회)을 담당

내 코드(`JpaMain.java`)에서 이 흐름 그대로:

```java
// persistence.xml 의 persistence-unit name="hello" 설정을 읽어
// EMF(싱글톤)를 생성
EntityManagerFactory emf = Persistence.createEntityManagerFactory("hello");
EntityManager em = emf.createEntityManager();   // 공장에서 EM 하나 생성
```

## 2. ⚠️ 반드시 지켜야 하는 3가지 주의사항

| 대상 | 규칙 | 이유 |
|------|------|------|
| **EntityManagerFactory** | 애플리케이션 전체에서 **딱 하나만** 생성해 공유 | 생성 비용이 크다(커넥션풀 등) |
| **EntityManager** | **쓰레드 간 공유 X**, 한 번 쓰고 버린다 | DB 커넥션과 밀접, 공유 시 동시성 문제 |
| **데이터 변경** | JPA의 모든 변경은 **트랜잭션 안에서** 실행 | 정합성 보장 |

내 코드에서 트랜잭션으로 감싸는 구조:

```java
EntityTransaction tx = em.getTransaction();
tx.begin();          // 트랜잭션 시작
try {
    // ... 데이터 작업 ...
    tx.commit();     // 성공 시 커밋
} catch (Exception e) {
    tx.rollback();   // 예외 시 롤백
} finally {
    em.close();      // 역순으로 자원 해제 (EM 먼저)
    emf.close();     // EMF 나중에
}
```

> 💡 자원 해제는 **생성의 역순**(EM → EMF). `finally`에서 반드시 닫아준다.

## 3. persistence.xml — JPA 설정 파일

- 위치는 **`/META-INF/persistence.xml`** 로 고정 (여기 없으면 못 찾음)
- `<persistence-unit name="hello">` 의 이름으로 `createEntityManagerFactory("hello")` 호출
- 속성 접두사로 표준/전용 구분:
  - **`jakarta.persistence`** 로 시작 → JPA 표준 속성 *(구버전은 `javax.persistence`)*
  - **`hibernate`** 로 시작 → 하이버네이트 전용 속성

> ⚠️ 강의 PDF는 `javax.persistence`를 쓰지만, 내 프로젝트는 **최신 버전(Hibernate 6.x)** 이라
> `jakarta.persistence` 를 사용한다. (Java EE → Jakarta EE 패키지 이동)

내 `persistence.xml` 필수 속성:

```xml
<!-- 필수 속성 -->
<property name="jakarta.persistence.jdbc.driver"   value="org.h2.Driver"/>
<property name="jakarta.persistence.jdbc.user"     value="sa"/>
<property name="jakarta.persistence.jdbc.password" value=""/>
<property name="jakarta.persistence.jdbc.url"      value="jdbc:h2:tcp://localhost/~/test"/>
<property name="hibernate.dialect" value="org.hibernate.dialect.H2Dialect"/>

<!-- 옵션 (개발 시 SQL 확인용) -->
<property name="hibernate.show_sql"       value="true"/>  <!-- 실행 SQL 콘솔 출력 -->
<property name="hibernate.format_sql"     value="true"/>  <!-- SQL 보기 좋게 정렬 -->
<property name="hibernate.use_sql_comments" value="true"/><!-- SQL에 주석 추가 -->
```

- JDBC URL `jdbc:h2:tcp://localhost/~/test` → H2 **TCP 서버 모드**로 접속
  (H2를 서버로 먼저 실행해둬야 함)

## 4. 데이터베이스 방언 (Dialect)

- JPA는 **특정 DB에 종속되지 않는다.** 하지만 DB마다 SQL 문법/함수가 조금씩 다름.

| 기능 | 표준/MySQL | Oracle |
|------|-----------|--------|
| 가변 문자 | `VARCHAR` | `VARCHAR2` |
| 문자열 자르기 | `SUBSTRING()` | `SUBSTR()` |
| 페이징 | `LIMIT` | `ROWNUM` |

- **방언(Dialect)** = SQL 표준을 지키지 않는 특정 DB만의 고유 기능을 JPA가 대신 처리해주는 것
- `hibernate.dialect` 속성으로 지정 → 하이버네이트가 그 DB에 맞는 SQL을 생성
  - H2 : `org.hibernate.dialect.H2Dialect`
  - Oracle : `org.hibernate.dialect.Oracle10gDialect`
  - MySQL : `org.hibernate.dialect.MySQL5InnoDBDialect`
- 하이버네이트는 40종 이상의 방언 지원 → **DB를 바꿔도 방언만 교체**하면 됨

## 5. 객체와 테이블 매핑 — @Entity / @Id

내 `Member.java`:

```java
import jakarta.persistence.Entity;
import jakarta.persistence.Id;

@Entity                 // JPA가 관리하는 엔티티로 인식 (클래스명 = 테이블명 매핑)
public class Member {

    @Id                 // DB의 PK(기본키)와 매핑
    private long id;
    private String name;

    // getter / setter
}
```

- **`@Entity`** : 이 클래스를 JPA가 관리 → 기본적으로 **클래스 이름 = 테이블 이름**
- **`@Id`** : 이 필드를 테이블의 **기본키(PK)** 로 매핑
- 위 매핑이 만드는 테이블:

```sql
create table Member (
    id   bigint not null,
    name varchar(255),
    primary key (id)
);
```

## 6. JPQL — 객체를 대상으로 하는 쿼리

### 왜 필요한가?
- JPA는 **엔티티 객체 중심**으로 개발한다. 단건 조회는 `em.find()`, 연관은 객체 그래프 탐색(`a.getB().getC()`)으로 충분.
- **문제는 검색.** "나이 18세 이상 회원 전체" 같이 조건이 붙으면?
  - 모든 데이터를 객체로 불러와 걸러내는 건 불가능 → 결국 **조건이 들어간 쿼리가 필요**.

### JPQL의 정체
- JPA가 제공하는 **객체 지향 쿼리 언어**. 한마디로 "객체 지향 SQL".
- **JPQL은 엔티티 객체를 대상**으로 쿼리 / **SQL은 테이블을 대상**으로 쿼리
- SQL을 추상화 → **특정 DB SQL에 의존하지 않음** (방언이 실제 SQL로 번역)
- 문법은 SQL과 유사: `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `JOIN` 지원

### 내 코드 — 페이징까지
```java
List<Member> result = em.createQuery("select m from Member as m", Member.class)
        .setFirstResult(5)   // 6번째부터 (0-based, 앞의 5개 건너뜀)
        .setMaxResults(8)    // 최대 8개
        .getResultList();
```

- `from Member` → **테이블이 아니라 엔티티 `Member`** 를 대상으로 조회
- **페이징**은 방언 덕분에 DB에 맞는 SQL(H2/MySQL은 LIMIT, Oracle은 ROWNUM)로 자동 변환
  → JPQL은 그대로 두고 DB만 바꿔도 페이징이 동작

> JPQL 상세 문법은 뒤 `10. 객체지향 쿼리 언어` 에서 깊게 다룸.

---

## ✅ 핵심 요약 한 줄 정리
1. **Persistence → EMF(하나, 공유) → EM(쓰다 버림)** 구동 흐름
2. **모든 데이터 변경은 트랜잭션 안에서**, 자원은 생성 역순으로 close
3. **persistence.xml**(META-INF)에서 설정, 최신 버전은 `jakarta.persistence` 접두사
4. **방언(Dialect)** 으로 DB 독립성 확보
5. **@Entity / @Id** 로 객체 ↔ 테이블 매핑
6. **JPQL** = 테이블이 아닌 **엔티티 객체**를 대상으로 하는 객체 지향 쿼리
