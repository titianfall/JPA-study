## 1. 영속성 컨텍스트란?

- 한마디로 **"엔티티를 영구 저장하는 환경"**. JPA를 이해하는 데 가장 중요한 용어.
- **논리적인 개념**이라 눈에 보이지 않는다. **`EntityManager`를 통해서** 영속성 컨텍스트에 접근한다.
- `em.persist(entity)` 는 사실 DB에 저장하는 게 아니라 **영속성 컨텍스트에 저장**하는 것.

```java
EntityManagerFactory emf = Persistence.createEntityManagerFactory("hello");
EntityManager em = emf.createEntityManager();
```

| 환경 | EntityManager ↔ 영속성 컨텍스트 |
|------|------------------------------|
| J2SE (지금 내 실습 환경) | **1 : 1** |
| J2EE / 스프링 컨테이너 | **N : 1** |

## 2. 엔티티의 생명주기 (Life Cycle)

```
              persist()
   [비영속] ───────────▶ [영속] ◀──── find() / JPQL (DB에서 조회)
    New                  Managed
                       │   ▲   │
        detach()/clear()   │   │ remove()
        /close() │         │   ▼
                 ▼      merge()  [삭제]
             [준영속]           Removed
             Detached
```

| 상태 | 설명 | 관련 메서드 |
|------|------|------------|
| **비영속 (new/transient)** | 영속성 컨텍스트와 전혀 관계없는 순수 객체 | `new Member()` |
| **영속 (managed)** | 영속성 컨텍스트가 관리하는 상태 | `em.persist()`, `em.find()` |
| **준영속 (detached)** | 영속 상태였다가 분리된 상태 | `em.detach()`, `em.clear()`, `em.close()` |
| **삭제 (removed)** | 삭제된 상태 | `em.remove()` |

내 코드의 비영속 → 영속 전환 예시:

```java
Member member = new Member();
member.setId(101L);
member.setName("HelloJPA");

System.out.println("=== BEFORE ===");
em.persist(member);
System.out.println("=== AFTER ===");
```

> ⚠️ `em.persist(member)` 를 호출해도 **BEFORE ~ AFTER 사이에 INSERT 쿼리가 나가지 않는다.**
> `persist` 는 "영속성 컨텍스트(1차 캐시)에 등록"일 뿐, DB 반영은 커밋(flush) 시점이다. → 아래 3-3.

## 3. 영속성 컨텍스트의 5가지 이점

> 1차 캐시 · 동일성 보장 · 쓰기 지연 · 변경 감지 · 지연 로딩

### 3-1. 1차 캐시 — 조회 시 SQL을 생략

- 영속성 컨텍스트 안에는 `@Id` 를 key로, 엔티티를 value로 갖는 **1차 캐시**가 있다.
- `em.find()` 는 **1차 캐시를 먼저 조회** → 있으면 그대로 반환(SQL 없음), 없으면 DB 조회 후 1차 캐시에 저장하고 반환.

```java
System.out.println("findMember1");
Member findMember1 = em.find(Member.class, 101L);
System.out.println("findMember2");
Member findMember2 = em.find(Member.class, 101L);
```

- 같은 트랜잭션에서 `persist` 직후 `find` 하면, 방금 1차 캐시에 넣은 객체를 그대로 꺼내므로 **SELECT 쿼리가 나가지 않는다.**

### 3-2. 동일성(identity) 보장

```java
System.out.println("isEqualTo = " + (findMember1 == findMember2));
```

- 같은 트랜잭션 안에서 같은 PK로 조회하면 **항상 같은 인스턴스**(`findMember1 == findMember2` → `true`).
- 1차 캐시로 **반복 가능한 읽기(REPEATABLE READ)** 등급의 격리 수준을 **애플리케이션 차원**에서 제공한다.

### 3-3. 트랜잭션을 지원하는 쓰기 지연 (transactional write-behind)

```java
Member member1 = new Member(150L, "A");
Member member2 = new Member(160L, "A");

em.persist(member1);
em.persist(member2);
System.out.println("================");
```

- `persist` 하면 INSERT SQL이 **쓰기 지연 SQL 저장소**에 쌓이기만 한다.
- **커밋(=flush)하는 순간** 모아둔 INSERT를 한 번에 DB로 전송 → 커밋. (위 `================` 이후 커밋 때 쿼리가 나감)

> 💡 내 `persistence.xml` 에 배치 옵션을 넣어 여러 INSERT를 묶어 보낼 수 있다:
> ```xml
> <property name="hibernate.jdbc.batch_size" value="10"/>
> ```
> ⚠️ 현재 내 파일엔 `hibernamte...` 오타가 있음 → `hibernate.jdbc.batch_size` 로 고쳐야 실제 적용된다.

### 3-4. 변경 감지 (Dirty Checking) — `update` 가 필요 없다

```java
Member memberA = em.find(Member.class, 150L);
memberA.setName("hi");
```

- 영속 엔티티는 **값만 바꿔도** JPA가 알아서 UPDATE 한다. `em.update()` 같은 메서드는 **존재하지 않는다.**
- 이미 영속 상태인 `memberA` 는 다시 `em.persist(memberA)` 를 호출할 필요가 없다.
- 동작 원리: 영속성 컨텍스트는 최초 조회 시점의 **스냅샷**을 떠둔다.
  → flush 시점에 **[엔티티 현재값 vs 스냅샷]** 을 비교 → 다르면 UPDATE SQL 자동 생성 → DB 반영.

### 3-5. 지연 로딩 (Lazy Loading)

- 연관 엔티티를 실제 사용하는 시점에 조회하는 기능. → 뒤 `08. 프록시와 연관관계 관리` 에서 상세히 다룸.

## 4. 플러시 (flush)

> **영속성 컨텍스트의 변경 내용을 데이터베이스에 반영**하는 것.

```java
Member member = new Member(200L, "member200");
em.persist(member);

em.flush();

System.out.println("================");
```

### 플러시가 하는 일
1. 변경 감지(Dirty Checking) 실행
2. 수정된 엔티티를 쓰기 지연 SQL 저장소에 등록
3. 쓰기 지연 SQL 저장소의 쿼리(등록/수정/삭제)를 DB에 전송

### 플러시가 발생하는 3가지 경우

| 방법 | 설명 |
|------|------|
| `em.flush()` | **직접 호출** |
| 트랜잭션 커밋 | 플러시 **자동 호출** |
| JPQL 쿼리 실행 | 플러시 **자동 호출** |

> 💡 **JPQL 실행 시 자동 flush 되는 이유**: `persist` 만 하고 아직 DB에 없는 엔티티를,
> 바로 이어지는 `select ... from Member` JPQL이 조회하지 못하면 곤란하기 때문. → 먼저 flush 해서 DB에 반영.

### ⚠️ 플러시에 대한 오해 정리
- 플러시는 영속성 컨텍스트를 **비우지 않는다.** (내용을 DB와 **동기화**만 함)
- 트랜잭션이라는 작업 단위가 중요 → **커밋 직전에만 동기화**하면 되므로 쓰기 지연이 가능한 것.
- `FlushModeType.AUTO`(기본): 커밋 + 쿼리 실행 시 flush / `FlushModeType.COMMIT`: 커밋할 때만 flush.

## 5. 준영속 상태 (detached)

- **영속 → 준영속**: 영속 상태 엔티티가 영속성 컨텍스트에서 **분리**된 상태.
- 준영속이 되면 **1차 캐시·동일성·변경 감지 등 영속성 컨텍스트의 기능을 못 쓴다.**

```java
Member member = em.find(Member.class, 150L);
member.setName("AAAAA");

em.clear();

Member member2 = em.find(Member.class, 150L);

System.out.println("=========================");
tx.commit();
```

### 준영속으로 만드는 3가지 방법

| 메서드 | 범위 | 효과 |
|--------|------|------|
| `em.detach(entity)` | **특정 엔티티 1개** | 그 엔티티만 준영속 → 변경 감지 X (UPDATE 안 나감) |
| `em.clear()` | **영속성 컨텍스트 전체** | 1차 캐시 통째로 초기화 → 이후 `find` 는 DB 재조회 |
| `em.close()` | 영속성 컨텍스트 종료 | 컨텍스트 자체를 닫음 |

> 내 코드 관찰 포인트:
> - `em.detach(member)` 를 쓰면 → `setName` 변경이 무시되어 **UPDATE 쿼리가 안 나간다** (SELECT만).
> - `em.clear()` 를 쓰면 → 캐시가 비어 `member2` 조회 시 **SELECT 쿼리가 한 번 더** 발생한다.

## 6. 내 실습 코드 기준 실행 흐름 정리

이번 실습 코드는 한 파일에서 블록을 하나씩 실행하면서 영속성 컨텍스트의 특징을 확인하는 방식이다.
각 실습 블록은 **SQL이 언제 나가는지**를 관찰하는 데 초점을 맞추면 이해하기 쉽다.

| 실습 블록 | 확인하려는 내용 | SQL 관찰 포인트 |
|-----------|----------------|----------------|
| `em.persist(member)` 전후 `BEFORE/AFTER` | `persist` 는 DB 저장이 아니라 영속 상태 전환 | `BEFORE` 와 `AFTER` 사이에 INSERT가 안 나감 |
| 같은 PK를 `em.find()` 두 번 호출 | 1차 캐시와 동일성 보장 | 첫 조회만 SELECT, 두 번째 조회는 SQL 없음 |
| `member1`, `member2` 를 연속 `persist` | 트랜잭션 쓰기 지연 | `persist` 시점이 아니라 커밋/flush 때 INSERT 전송 |
| 조회한 `memberA.setName("hi")` | 변경 감지 | `persist` 재호출 없이 커밋 시 UPDATE |
| `em.flush()` 직접 호출 | 커밋 전 DB 동기화 | `flush()` 줄에서 INSERT/UPDATE가 먼저 나감 |
| `em.detach(member)` 또는 `em.clear()` | 준영속 상태 | 관리 대상에서 빠지면 변경 감지 불가 |

### 현재 켜져 있는 코드의 흐름

```java
Member member = em.find(Member.class, 150L);
member.setName("AAAAA");

em.clear();

Member member2 = em.find(Member.class, 150L);

System.out.println("=========================");
tx.commit();
```

1. `em.find(Member.class, 150L)` 로 DB에서 회원을 조회하고 1차 캐시에 올린다.
2. `member.setName("AAAAA")` 로 값을 바꾸지만, 아직 SQL이 나간 것은 아니다.
3. `em.clear()` 로 영속성 컨텍스트 전체를 비운다.
4. 그래서 `member` 는 더 이상 관리 대상이 아니고, 방금 바꾼 `"AAAAA"` 변경은 변경 감지 대상에서 제외된다.
5. 다시 `em.find(Member.class, 150L)` 를 호출하면 1차 캐시가 비어 있으므로 SELECT가 한 번 더 나간다.
6. `tx.commit()` 시점에는 현재 관리 중인 `member2` 에 별도 변경이 없으므로 UPDATE가 나가지 않는다.

> 즉, 지금 실행 중인 코드에서 핵심은 **`setName` 후 `clear()` 하면 변경 감지가 끊긴다**는 점이다.
> `clear()` 를 주석 처리하면 커밋 시점에 변경 감지가 동작해서 UPDATE가 나간다.

### `detach` 와 `clear` 의 차이

| 코드 | 범위 | 결과 |
|------|------|------|
| `em.detach(member)` | 특정 엔티티 하나만 분리 | `member` 의 변경만 추적하지 않음 |
| `em.clear()` | 영속성 컨텍스트 전체 초기화 | 캐시가 모두 비어서 이후 조회는 DB를 다시 조회 |

둘 다 준영속 상태를 만들 수 있지만, 실습 로그에서는 `clear()` 쪽이 더 눈에 띈다.
왜냐하면 `member2` 를 다시 조회할 때 **1차 캐시가 사라졌기 때문에 SELECT가 다시 발생**하기 때문이다.

## 7. 동작 보정 메모

실습 코드의 동작을 더 정확히 이해하려면 아래 표현을 기준으로 잡으면 된다.

### `em.find()` 와 1차 캐시

`find` 는 DML 버퍼에 넣는 동작이 아니다.
정확히는 DB에서 조회한 엔티티를 **영속성 컨텍스트의 1차 캐시에 저장하고, 스냅샷도 함께 보관**한다.

```java
Member memberA = em.find(Member.class, 150L);
memberA.setName("hi");
```

쓰기 지연 SQL 저장소에 UPDATE가 들어가는 시점은 `setName` 직후가 아니라 **flush 시점의 변경 감지 이후**다.

### `em.persist(memberA)` 를 다시 호출하지 않는 이유

이미 `em.find()` 로 조회한 `memberA` 는 영속 상태다.
따라서 컬렉션에서 객체를 꺼내 값을 바꾼 뒤 다시 넣지 않는 것처럼, 영속 엔티티도 값을 바꾼 뒤 다시 `persist` 할 필요가 없다.

```java
Member memberA = em.find(Member.class, 150L);
memberA.setName("hi");
```

### `flush` 와 `commit` 의 관계

- `flush` 는 SQL을 DB에 보내 **동기화**한다.
- `commit` 은 트랜잭션을 실제로 **확정**한다.
- JPA는 `tx.commit()` 직전에 자동으로 `flush` 를 호출한다.

그래서 직접 `em.flush()` 를 호출하면 커밋 전에도 SQL 로그를 먼저 볼 수 있지만,
트랜잭션을 커밋하지 않고 롤백하면 DB에 최종 반영되지는 않는다.

### Hibernate 6 실습 환경 메모

강의 PDF는 예전 `javax.persistence` 기준 설명이 섞여 있지만, 현재 실습 코드는 `jakarta.persistence` 패키지를 사용한다.
핵심 개념인 영속성 컨텍스트, 1차 캐시, 쓰기 지연, 변경 감지, 플러시는 동일하게 이해하면 된다.

## ✅ 핵심 요약 한 줄 정리
1. **영속성 컨텍스트** = 엔티티를 영구 저장하는 환경, `EntityManager` 로 접근하는 논리적 개념
2. 생명주기: **비영속 → (persist) → 영속 → (detach/clear/close) → 준영속 / (remove) → 삭제**
3. **1차 캐시**로 조회 SQL 생략, **동일성(`==`) 보장**
4. **쓰기 지연**: persist는 저장소에 쌓고 **커밋(flush) 때 한 번에** INSERT
5. **변경 감지(Dirty Checking)**: 값만 바꾸면 UPDATE 자동, `em.update()` 는 없다 (스냅샷 비교)
6. **flush** = DB 동기화(비우지 않음), 발생 = `em.flush()` / 커밋 / JPQL 실행
7. **준영속**되면 영속성 컨텍스트의 혜택을 못 받는다 (detach·clear·close)
