# 06. 다양한 연관관계 매핑 - 다중성별 매핑 선택법

> 강의자료 `JPA 기본편 - 강의자료/06. 다양한 연관관계 매핑.pdf` 기반
> 5장에서 만든 `study/jpabook` 연관관계 코드를 기준으로, 어떤 매핑을 선택하고 피해야 하는지 정리

---

## 0. 이 장의 지도

연관관계 매핑을 볼 때는 항상 아래 3가지를 같이 본다.

| 고려 사항 | 확인 질문 | 핵심 |
|-----------|-----------|------|
| 다중성 | N:1, 1:N, 1:1, N:M 중 무엇인가? | `@ManyToOne`, `@OneToMany`, `@OneToOne`, `@ManyToMany` |
| 방향 | 객체에서 어느 쪽으로 탐색해야 하는가? | 참조 필드가 있는 쪽으로만 객체 그래프 탐색 가능 |
| 연관관계의 주인 | 어떤 참조가 외래 키를 관리하는가? | 외래 키가 있는 테이블과 가까운 쪽을 주인으로 둔다 |

테이블은 외래 키 하나로 양쪽 조인이 가능해서 "방향"이라는 개념이 거의 없다. 반대로 객체는 참조 필드가 있는 방향으로만 이동할 수 있다.

객체의 양방향은 사실 **단방향 참조 2개**다. 그래서 둘 중 하나만 외래 키를 관리해야 하고, 그 관리 주체를 **연관관계의 주인**이라고 부른다.

---

## 1. 내 코드의 현재 위치

5장까지의 `study/jpabook` 코드는 이미 대부분 **다대일 주인 + 필요하면 일대다 반대편** 구조로 정리되어 있다.

| 관계 | 주인 | 반대편 | 테이블 FK |
|------|------|--------|-----------|
| `Member` - `Order` | `Order.member` | `Member.orders` | `ORDERS.MEMBER_ID` |
| `Order` - `OrderItem` | `OrderItem.order` | `Order.orderItems` | `ORDER_ITEM.ORDER_ID` |
| `Item` - `OrderItem` | `OrderItem.item` | 없음 | `ORDER_ITEM.ITEM_ID` |

예를 들어 `OrderItem`은 주문과 상품의 외래 키 값을 직접 들고 있지 않고, 전부 참조로 바뀌어 있다.

```java
@Entity
public class OrderItem {
    @ManyToOne
    @JoinColumn(name = "ORDER_ID")
    private Order order;

    @ManyToOne
    @JoinColumn(name = "ITEM_ID")
    private Item item;
}
```

이 모양이 6장에서 계속 반복되는 기본형이다. **외래 키가 있는 N쪽에 `@ManyToOne`을 두고, 반대 방향 조회가 필요할 때만 `@OneToMany(mappedBy = "...")`를 추가한다.**

## 2. 다대일 N:1 - 가장 기본이고 가장 많이 쓴다

### 다대일 단방향

회원과 주문을 보면 주문 N개가 회원 1명에게 속한다. 테이블 FK는 `ORDERS.MEMBER_ID`에 있으므로 `Order.member`가 주인이다.

```java
@Entity
@Table(name = "ORDERS")
public class Order {
    @Id @GeneratedValue
    @Column(name = "ORDER_ID")
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "MEMBER_ID")
    private Member member;
}
```

- `@ManyToOne`: 현재 엔티티가 N, 상대 엔티티가 1이다.
- `@JoinColumn(name = "MEMBER_ID")`: 이 참조를 `ORDERS.MEMBER_ID` FK와 매핑한다.
- 실무에서는 `@ManyToOne`의 기본값인 EAGER를 그대로 두지 말고 보통 `LAZY`로 둔다.

### 다대일 양방향

회원에서 주문 목록을 탐색해야 하면 반대편에 컬렉션을 추가한다.

```java
@Entity
public class Member {
    @OneToMany(mappedBy = "member")
    private List<Order> orders = new ArrayList<>();
}
```

여기서 `mappedBy = "member"`는 `Order.member`가 주인이라는 뜻이다. `Member.orders`는 FK를 바꾸지 않고 조회용으로만 쓰인다.

현재 내 코드도 같은 구조다.

```java
// Member.java
@OneToMany(mappedBy = "member")
private List<Order> orders = new ArrayList<>();

// Order.java
@ManyToOne
@JoinColumn(name = "MEMBER_ID")
private Member member;
```

정리하면 다대일은 이렇게 잡으면 된다.

1. FK가 있는 N쪽에 `@ManyToOne`을 둔다.
2. 그 필드가 연관관계의 주인이다.
3. 반대 방향 탐색이 필요하면 1쪽에 `@OneToMany(mappedBy = "주인필드명")`을 추가한다.

## 3. 일대다 1:N - 가능하지만 권장하지 않는다

### 일대다 단방향

일대다 단방향은 1쪽 객체가 N쪽 테이블의 FK를 관리하는 구조다.

```java
@Entity
public class Team {
    @OneToMany
    @JoinColumn(name = "TEAM_ID")
    private List<Member> members = new ArrayList<>();
}
```

테이블 구조는 여전히 `MEMBER.TEAM_ID`가 FK다. 그런데 객체에서는 `Team.members`가 그 FK를 관리한다.

이 구조가 어색한 이유:

- 엔티티가 관리하는 FK가 자기 테이블이 아니라 **반대편 테이블**에 있다.
- `Member`를 INSERT한 뒤 `TEAM_ID`를 맞추기 위해 UPDATE SQL이 추가로 나갈 수 있다.
- `@JoinColumn`을 생략하면 FK 방식이 아니라 중간 조인 테이블 방식으로 만들어질 수 있다.

예상 SQL 흐름은 이런 식이다.

```sql
insert into Member ...
update Member set TEAM_ID = ? where MEMBER_ID = ?
```

그래서 강의 결론은 단순하다.

> 일대다 단방향보다 다대일 양방향을 사용하자.

### 일대다 양방향처럼 쓰는 방법

JPA에 공식적인 "일대다 양방향" 매핑은 없다. 억지로 만들려면 N쪽에 읽기 전용 `@ManyToOne`을 둔다.

```java
@Entity
public class Member {
    @ManyToOne
    @JoinColumn(name = "TEAM_ID", insertable = false, updatable = false)
    private Team team;
}
```

`insertable = false`, `updatable = false`라서 이 필드는 FK를 변경하지 못한다. 읽기 전용으로 양방향처럼 보이게 만드는 방법이다.

하지만 이건 구조가 헷갈리기 쉽다. 처음부터 N쪽을 주인으로 잡는 **다대일 양방향**이 더 자연스럽다.

## 4. 일대일 1:1 - FK 위치를 선택할 수 있다

일대일은 양쪽 모두 하나씩만 연결된다. 그래서 주 테이블에 FK를 둘 수도 있고, 대상 테이블에 FK를 둘 수도 있다. 단, FK에는 유니크 제약조건이 필요하다.

예시: 회원과 사물함

| 선택 | FK 위치 | 특징 |
|------|---------|------|
| 주 테이블에 FK | `MEMBER.LOCKER_ID` | 객체지향 개발자가 선호, JPA 매핑 편리 |
| 대상 테이블에 FK | `LOCKER.MEMBER_ID` | DB 설계 관점에서 선호될 수 있음, 나중에 1:N으로 바꾸기 쉬움 |

### 주 테이블에 외래 키 - 단방향

`Member`가 `Locker`를 참조하고, `MEMBER.LOCKER_ID`가 FK인 구조다.

```java
@Entity
public class Member {
    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "LOCKER_ID", unique = true)
    private Locker locker;
}
```

`@ManyToOne` 단방향과 거의 똑같이 생각하면 된다. 차이는 다중성이 1:1이고, FK에 유니크 제약조건이 들어간다는 점이다.

### 주 테이블에 외래 키 - 양방향

반대편에서 회원을 탐색해야 하면 `mappedBy`를 사용한다.

```java
@Entity
public class Locker {
    @OneToOne(mappedBy = "locker")
    private Member member;
}
```

외래 키가 있는 `Member.locker`가 주인이고, `Locker.member`는 읽기 전용 반대편이다.

### 대상 테이블에 외래 키

이번에는 `LOCKER.MEMBER_ID`가 FK인 구조다.

```java
@Entity
public class Locker {
    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "MEMBER_ID", unique = true)
    private Member member;
}
```

반대편은 이렇게 둔다.

```java
@Entity
public class Member {
    @OneToOne(mappedBy = "member")
    private Locker locker;
}
```

주의할 점:

- 대상 테이블에 FK가 있는 **단방향** 관계는 JPA가 지원하지 않는다.
- 대상 테이블에 FK가 있는 구조는 양방향으로 매핑해야 한다.
- 프록시 한계 때문에 지연 로딩으로 설정해도 즉시 로딩처럼 동작할 수 있다. 이 내용은 8장 프록시에서 다시 중요해진다.

일대일 선택 기준은 이렇게 잡으면 된다.

| 기준 | 주 테이블 FK | 대상 테이블 FK |
|------|--------------|----------------|
| 객체 탐색 | 주 객체가 대상 객체를 바로 찾기 좋음 | 반대편 FK라 객체 관점에서 어색할 수 있음 |
| JPA 매핑 | 편함 | 단방향 미지원, 양방향 필요 |
| 데이터 확인 | 주 테이블만 봐도 대상 존재 여부 확인 가능 | 대상 테이블을 봐야 확인 가능 |
| 확장성 | 값이 없으면 FK null 허용 필요 | 나중에 1:N으로 바꿀 때 테이블 구조 유지 가능 |

## 5. 다대다 N:M - 문법은 있지만 실무에서는 피한다

관계형 데이터베이스는 정규화된 테이블 2개만으로 다대다를 표현할 수 없다. 반드시 연결 테이블이 필요하다.

객체는 컬렉션으로 다대다처럼 보이게 만들 수 있다.

```java
@Entity
public class Member {
    @ManyToMany
    @JoinTable(name = "MEMBER_PRODUCT")
    private List<Product> products = new ArrayList<>();
}
```

하지만 실무에서는 `@ManyToMany`를 거의 쓰지 않는다.

이유:

- 연결 테이블이 단순히 `MEMBER_ID`, `PRODUCT_ID`만 가지는 경우가 드물다.
- 주문 수량, 주문 시간, 가격, 상태 같은 추가 컬럼이 들어온다.
- 연결 테이블을 엔티티로 다루지 않으면 이런 필드를 자연스럽게 담을 수 없다.

그래서 연결 테이블을 엔티티로 승격한다.

```java
@Entity
public class MemberProduct {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "MEMBER_ID")
    private Member member;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "PRODUCT_ID")
    private Product product;

    private int orderAmount;
    private LocalDateTime orderDate;
}
```

그리고 양쪽은 일대다, 다대일로 풀어낸다.

```java
@Entity
public class Member {
    @OneToMany(mappedBy = "member")
    private List<MemberProduct> memberProducts = new ArrayList<>();
}

@Entity
public class Product {
    @OneToMany(mappedBy = "product")
    private List<MemberProduct> memberProducts = new ArrayList<>();
}
```

내 `jpabook` 코드의 `OrderItem`이 바로 이 생각과 같다.

`Order`와 `Item`은 개념적으로 N:M처럼 보일 수 있지만, 실제 주문 상품에는 `orderPrice`, `count`가 필요하다. 그래서 `OrderItem`이라는 연결 엔티티를 만들고 `OrderItem.order`, `OrderItem.item`을 각각 `@ManyToOne`으로 매핑했다.

```java
@Entity
public class OrderItem {
    @ManyToOne
    @JoinColumn(name = "ORDER_ID")
    private Order order;

    @ManyToOne
    @JoinColumn(name = "ITEM_ID")
    private Item item;

    private int orderPrice;
    private int count;
}
```

## 6. 실전 예제 3 - 배송, 카테고리 추가

6장 실전 예제에서는 기존 주문 도메인에 배송과 카테고리가 추가된다.

| 추가 관계 | 다중성 | 테이블 구조 | 객체 매핑 |
|-----------|--------|-------------|-----------|
| `Order` - `Delivery` | 1:1 | `ORDERS.DELIVERY_ID` | `Order.delivery`가 주인 |
| `Category` - `Item` | N:M | `CATEGORY_ITEM` 연결 테이블 | 강의 문법상 `@ManyToMany`, 실무식은 연결 엔티티 권장 |
| `Category` - `Category` | N:1 자기참조 | `CATEGORY.PARENT_ID` | `Category.parent`가 주인 |

### 주문 - 배송

ERD에서 `ORDERS` 테이블에 `DELIVERY_ID`가 있으므로 `Order.delivery`가 주인이다.

```java
@Entity
@Table(name = "ORDERS")
public class Order {
    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "DELIVERY_ID")
    private Delivery delivery;
}
```

반대 방향 탐색이 필요하면 `Delivery`에 `mappedBy`를 둔다.

```java
@Entity
public class Delivery {
    @OneToOne(mappedBy = "delivery")
    private Order order;

    private String city;
    private String street;
    private String zipcode;
    private DeliveryStatus status;
}
```

### 카테고리 자기참조

카테고리는 부모 카테고리 하나를 가질 수 있고, 자식 카테고리는 여러 개 가질 수 있다. 테이블 FK는 `CATEGORY.PARENT_ID`이므로 `parent`가 주인이다.

```java
@Entity
public class Category {
    @Id @GeneratedValue
    @Column(name = "CATEGORY_ID")
    private Long id;

    private String name;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "PARENT_ID")
    private Category parent;

    @OneToMany(mappedBy = "parent")
    private List<Category> child = new ArrayList<>();

    public void addChildCategory(Category child) {
        this.child.add(child);
        child.setParent(this);
    }
}
```

자기참조도 원리는 똑같다. 같은 엔티티를 바라볼 뿐, FK가 있는 `parent` 쪽이 주인이다.

### 상품 - 카테고리

강의 예제에서는 `CATEGORY_ITEM` 연결 테이블을 `@ManyToMany`로 매핑할 수 있다.

```java
@Entity
public class Category {
    @ManyToMany
    @JoinTable(
        name = "CATEGORY_ITEM",
        joinColumns = @JoinColumn(name = "CATEGORY_ID"),
        inverseJoinColumns = @JoinColumn(name = "ITEM_ID")
    )
    private List<Item> items = new ArrayList<>();
}
```

반대편을 열면 `Item`에는 `mappedBy`를 둔다.

```java
@Entity
public class Item {
    @ManyToMany(mappedBy = "items")
    private List<Category> categories = new ArrayList<>();
}
```

다만 이건 문법 확인용에 가깝다. `CATEGORY_ITEM`에 정렬 순서, 등록자, 노출 여부 같은 컬럼이 추가될 가능성이 있으면 아래처럼 연결 엔티티로 승격하는 편이 좋다.

```java
@Entity
public class CategoryItem {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "CATEGORY_ID")
    private Category category;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "ITEM_ID")
    private Item item;
}
```

## 7. 주요 어노테이션 속성 정리

### @JoinColumn

외래 키 컬럼을 매핑할 때 사용한다.

| 속성 | 의미 | 기본값 |
|------|------|--------|
| `name` | 매핑할 FK 이름 | `필드명_참조테이블PK컬럼명` |
| `referencedColumnName` | FK가 참조하는 대상 테이블 컬럼명 | 대상 테이블 PK 컬럼명 |
| `foreignKey` | DDL 생성 시 FK 제약조건 이름 지정 | 자동 생성 |
| `unique`, `nullable`, `insertable`, `updatable` | `@Column`과 비슷한 제약/쓰기 옵션 | 기본값은 속성별로 다름 |

대부분은 `name`을 가장 많이 쓴다.

```java
@JoinColumn(name = "MEMBER_ID")
private Member member;
```

### @ManyToOne

다대일 관계 매핑이다.

| 속성 | 의미 | 기본값 |
|------|------|--------|
| `optional` | false면 연관 엔티티가 반드시 있어야 함 | true |
| `fetch` | 글로벌 페치 전략 | EAGER |
| `cascade` | 영속성 전이 | 없음 |
| `targetEntity` | 연관 엔티티 타입 직접 지정 | 거의 사용하지 않음 |

실무 기본 습관은 이렇게 잡는 편이 좋다.

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "MEMBER_ID")
private Member member;
```

### @OneToMany

일대다 관계 매핑이다.

| 속성 | 의미 | 기본값 |
|------|------|--------|
| `mappedBy` | 반대편 주인 필드명 | 없음 |
| `fetch` | 글로벌 페치 전략 | LAZY |
| `cascade` | 영속성 전이 | 없음 |
| `targetEntity` | 컬렉션 대상 타입 직접 지정 | 거의 사용하지 않음 |

`mappedBy`가 있으면 주인이 아니다.

```java
@OneToMany(mappedBy = "order")
private List<OrderItem> orderItems = new ArrayList<>();
```

주의할 점은 `cascade`가 연관관계의 주인을 바꾸는 기능이 아니라는 것이다. FK를 누가 관리하는지는 여전히 `mappedBy`와 `@JoinColumn`으로 결정된다.

---

## 핵심 요약

1. 연관관계 매핑은 다중성, 방향, 주인을 같이 본다.
2. 다대일은 가장 많이 쓰는 기본형이고, FK가 있는 N쪽이 주인이다.
3. 일대다는 가능하지만 FK가 반대편 테이블에 있어 UPDATE가 추가될 수 있으므로 다대일 양방향을 우선한다.
4. 일대일은 FK를 주 테이블이나 대상 테이블 중 하나에 둘 수 있고, FK에는 유니크 제약조건이 필요하다.
5. 다대다는 `@ManyToMany` 문법이 있어도 실무에서는 연결 엔티티로 풀어내는 것이 안전하다.
6. 배송은 `Order.delivery`가 주인, 카테고리 자기참조는 `Category.parent`가 주인, 상품-카테고리 N:M은 실무에서는 `CategoryItem` 같은 연결 엔티티 후보로 본다.
