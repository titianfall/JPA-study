## 1. 왜 프록시가 필요한가?

Member를 조회할 때 Team도 항상 함께 조회해야 할까?

```java
// 회원과 팀을 함께 출력
public void printUserAndTeam(String memberId) {
    Member member = em.find(Member.class, memberId);
    Team team = member.getTeam();
    System.out.println("회원 이름: " + member.getUsername());
    System.out.println("소속팀: " + team.getName());
}

// 회원만 출력 — Team은 쓰지도 않는데 같이 조회하면 낭비
public void printUser(String memberId) {
    Member member = em.find(Member.class, memberId);
    System.out.println("회원 이름: " + member.getUsername());
}
```

비즈니스 로직에 따라 연관 엔티티가 필요할 수도, 아닐 수도 있다.
JPA는 **"실제 사용하는 시점까지 DB 조회를 미루는"** 장치로 **프록시**를 제공하고, 이를 기반으로 **지연 로딩**을 구현한다.

## 2. 프록시 기초 — `em.find()` vs `em.getReference()`

| 메서드 | 동작 |
|--------|------|
| `em.find()` | DB를 **즉시 조회**해서 실제 엔티티 객체 반환 |
| `em.getReference()` | DB 조회를 **미루고** 가짜(프록시) 객체 반환 — 이 시점엔 SELECT 없음 |

```java
Member member = em.getReference(Member.class, 1L); // SELECT 안 나감
member.getName();                                  // 이 순간 SELECT 실행 (프록시 초기화)
```

### 프록시 객체의 구조

- 실제 엔티티 클래스를 **상속**받아 만들어진다 → 겉모양이 같아서 사용하는 쪽은 진짜인지 프록시인지 구분 없이 쓰면 됨(이론상).
- 내부에 실제 객체를 가리키는 **`target` 참조**를 보관하고, 메서드가 호출되면 실제 객체에 **위임(delegate)** 한다.

### 프록시 초기화 과정

```
member.getName() 호출
   │ (target이 비어 있으면)
   ▼
영속성 컨텍스트에 초기화 요청 ──▶ DB 조회 ──▶ 실제 Entity 생성
   │
   ▼
프록시의 target이 실제 엔티티를 가리킴 → 이후 호출은 전부 위임
```

⚠️ 초기화돼도 **프록시 객체가 실제 엔티티로 바뀌는 게 아니다.** 프록시는 그대로 두고 내부 target만 채워진다.

## 3. 프록시의 특징 (시험 포인트)

| 특징 | 설명 |
|------|------|
| 초기화는 최초 1회 | 처음 사용할 때 한 번만 초기화된다 |
| 타입 비교 주의 | 원본을 상속한 클래스라 `==` 비교 실패 가능 → **`instanceof`** 사용 |
| 영속성 컨텍스트 우선 | 이미 1차 캐시에 실제 엔티티가 있으면 `getReference()`도 **실제 엔티티**를 반환 (반대로 프록시를 먼저 받았으면 `em.find()`도 프록시 반환 — JPA는 같은 트랜잭션 안에서 `==` 동일성을 보장하기 때문) |
| 준영속 상태 위험 | 영속성 컨텍스트가 닫힌(detach/close) 뒤 초기화하면 **`org.hibernate.LazyInitializationException`** 발생 — 실무에서 정말 자주 만나는 예외 |

### 프록시 확인 유틸리티

```java
// 초기화 여부 확인
emf.getPersistenceUnitUtil().isLoaded(entity);

// 프록시 클래스인지 확인 (..HibernateProxy.. 가 찍힘)
entity.getClass().getName();

// 강제 초기화 (하이버네이트 전용)
org.hibernate.Hibernate.initialize(entity);
```

💡 JPA 표준에는 강제 초기화가 없다. 표준 범위에서는 `member.getName()`처럼 **직접 값을 사용해서** 초기화한다.

내 `jpabook/JpaMain.java`의 아래 패턴은 프록시 실험의 필수 준비 작업이다.
1차 캐시에 엔티티가 남아 있으면 `getReference()`가 실제 엔티티를 돌려줘서 프록시 동작을 관찰할 수 없기 때문이다.

```java
em.persist(book);

em.flush();   // SQL을 DB에 먼저 반영
em.clear();   // 영속성 컨텍스트 초기화 → 이후 조회는 DB에서 새로
```

## 4. 즉시 로딩(EAGER)과 지연 로딩(LAZY)

프록시 메커니즘을 연관관계에 적용한 것이 fetch 전략이다.

| 전략 | 동작 | 연관 필드의 상태 |
|------|------|------------------|
| `FetchType.LAZY` | 연관 엔티티를 **실제 사용할 때** 별도 SELECT | **프록시**가 들어감 |
| `FetchType.EAGER` | 주 엔티티 조회 시 **조인해서 즉시** 함께 조회 | 실제 엔티티가 들어감 |

```java
@ManyToOne(fetch = FetchType.LAZY) // 지연 로딩
@JoinColumn(name = "TEAM_ID")
private Team team;
```

```java
Member member = em.find(Member.class, 1L); // Member만 SELECT
Team team = member.getTeam();              // 아직 프록시, 쿼리 없음
team.getName();                            // 이 시점에 Team SELECT (초기화)
```

### 기본값 주의 ⚠️

| 애노테이션 | 기본 fetch |
|------------|-----------|
| `@ManyToOne`, `@OneToOne` | **EAGER** ← 함정 |
| `@OneToMany`, `@ManyToMany` | LAZY |

### 실무 지침 — 즉시 로딩을 쓰지 마라

- 즉시 로딩은 **예상하지 못한 SQL**(거대한 조인)을 만든다.
- JPQL에서 **N+1 문제**를 일으킨다: `select m from Member m`은 SQL로 그대로 번역돼 Member만 조회하는데, EAGER인 Team을 채우려고 **회원 수(N)만큼 Team 조회 쿼리가 추가로** 나간다.
- 이론상 "자주 함께 쓰면 EAGER"라고 하지만, **실무에서는 전부 LAZY로 두고** 함께 조회가 필요한 화면은 **JPQL fetch join / 엔티티 그래프**로 해결한다(10장).

## 5. 내 코드에 적용 — fetch 전략 점검 (✅ 적용 완료, 커밋 `0ea92ab`)

원래 `jpabook` 도메인은 fetch를 명시하지 않아 `@ManyToOne`/`@OneToOne`이 전부 **암묵적 EAGER**였다. 8장에서 전부 LAZY로 명시했다.

| 파일 | 연관관계 | 변경 전(기본값) | 변경 후 |
|------|----------|-----------------|---------|
| `Order.java` | `member`, `delivery` | EAGER | ✅ `fetch = FetchType.LAZY` |
| `OrderItem.java` | `order`, `item` | EAGER | ✅ `fetch = FetchType.LAZY` |
| `Category.java` | `parent` (`@ManyToOne`) | EAGER | ✅ `fetch = FetchType.LAZY` |
| `CategoryItem.java` | `category`, `item` | EAGER | ✅ `fetch = FetchType.LAZY` |

```java
// Order.java — 적용 후
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "MEMBER_ID")
private Member member;
```

💡 `@OneToMany`(예: `Item.categoryItems`, `Category.child`)는 기본이 LAZY라 그대로 둬도 되지만, 팀 컨벤션에 따라 명시하기도 한다.

## 6. 영속성 전이 — CASCADE

**부모를 persist/remove 할 때 자식도 함께** 처리하고 싶을 때 쓴다.

```java
@OneToMany(mappedBy = "parent", cascade = CascadeType.PERSIST)
private List<Child> childList = new ArrayList<>();
```

```java
Parent parent = new Parent();
parent.addChild(child1);
parent.addChild(child2);

em.persist(parent);   // 이 한 줄로 child1, child2까지 함께 persist
```

### ⚠️ CASCADE는 연관관계 매핑과 무관하다

| 구분 | 역할 |
|------|------|
| `@JoinColumn` / `mappedBy` | FK 관리 주체(연관관계 주인) 결정 |
| `cascade` | 영속성 **상태 변화를 전파**하는 편의 기능일 뿐 |

cascade를 걸어도 연관관계 주인이 바뀌거나 조회 방식이 달라지지 않는다.

### 종류

| 타입 | 의미 | 비고 |
|------|------|------|
| `ALL` | 아래 전부 | 생명주기를 완전히 맡길 때 |
| `PERSIST` | 저장 전이 | 실무에서 주로 사용 |
| `REMOVE` | 삭제 전이 | |
| `MERGE` / `REFRESH` / `DETACH` | 각 연산 전이 | 자주 안 씀 |

### 사용 조건

- 자식의 **소유자(부모)가 하나**일 때만 — 예: `Order` ↔ `Delivery`처럼 단일 소유.
- `Delivery`를 다른 엔티티도 참조한다면 cascade를 걸면 안 된다(의도치 않은 저장/삭제 전파).

## 7. 고아 객체 — orphanRemoval

부모와의 **연관관계가 끊어진 자식을 자동으로 DELETE** 하는 기능.

```java
@OneToMany(mappedBy = "parent", orphanRemoval = true)
private List<Child> childList = new ArrayList<>();
```

```java
Parent parent1 = em.find(Parent.class, id);
parent1.getChildList().remove(0);   // 컬렉션에서 제거만 했는데
// → DELETE FROM CHILD WHERE ID = ?  실행
```

### CASCADE와 비교

| 항목 | `cascade = REMOVE` | `orphanRemoval = true` |
|------|--------------------|------------------------|
| 부모 삭제 시 자식 삭제 | O | O (고아가 되므로) |
| **컬렉션에서 제거 시** 자식 삭제 | X | **O** |
| 사용 가능 위치 | 모든 연관관계 | `@OneToOne`, `@OneToMany`만 |

⚠️ 둘 다 "참조하는 곳이 하나뿐 + 특정 엔티티가 개인 소유"일 때만 사용한다.

### 생명주기 관리 — `CascadeType.ALL + orphanRemoval = true`

두 옵션을 함께 켜면 자식의 생명주기를 **부모가 완전히 관리**한다.
자식에 대한 `em.persist()`, `em.remove()`를 직접 호출할 일이 없어지고, 부모 컬렉션 조작만으로 저장/삭제가 이뤄진다.
→ DDD의 **Aggregate Root** 패턴 구현에 유용.

## 8. 실전 예제 5 — 연관관계 관리 (✅ 적용 완료, 커밋 `0ea92ab`)

강의의 결론은 두 가지고, 내 `Order.java`에 그대로 반영했다.

1. **모든 연관관계를 지연 로딩으로** (`@ManyToOne`, `@OneToOne`에 `fetch = FetchType.LAZY` 명시)
2. **`Order → Delivery`, `Order → OrderItem`에 `CascadeType.ALL`** — 주문 저장 시 배송·주문상품이 함께 저장되도록

```java
// Order.java — 적용 후
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "MEMBER_ID")
private Member member;

@OneToOne(fetch = FetchType.LAZY, cascade = CascadeType.ALL)
@JoinColumn(name = "DELIVERY_ID")
private Delivery delivery;

@OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
private List<OrderItem> orderItems = new ArrayList<>();
```

> 💡 `Order → OrderItem` cascade를 위해 기존에 없던 `orderItems` 컬렉션(양방향)을 새로 열었다.
> `Delivery`에 cascade를 걸 수 있는 근거: Delivery를 참조하는 엔티티가 `Order` 하나뿐이고, 배송은 주문 없이 존재할 의미가 없다(단일 소유 + 생명주기 종속).

## 9. 실습 기록 — 커밋별 정리 (`ex1-hello-jpa`)

Member ↔ Team(다대일 양방향), Parent ↔ Child(일대다 양방향)로 직접 실험한 내용.

### 10-1. `find` vs `getReference` + 타입 체크 (`fb7dada`)

```java
Member findMember = em.find(Member.class, member.getId());           // 진짜 Member
Member findMember2 = em.getReference(Member.class, member2.getId()); // 프록시

findMember.getClass();  // class hellojpa.Member
findMember2.getClass(); // class hellojpa.Member$HibernateProxy$xxxx — 상속받은 자식 클래스

// 그래서 클래스 == 비교는 실패, 타입 체크는 instanceof
findMember2 instanceof Member; // true
```

### 10-2. 영속성 컨텍스트에 이미 엔티티가 있으면? (`874e781`, `9ff0fb7`)

```java
// find 먼저 → getReference도 실제 엔티티 반환
Member m1 = em.find(Member.class, 1L);              // hellojpa.Member
Member reference = em.getReference(Member.class, 1L); // hellojpa.Member (프록시 아님!)
System.out.println(m1 == reference); // true

// 반대로 getReference 먼저 → find도 프록시 반환
Member ref = em.getReference(Member.class, id);  // Proxy
Member find = em.find(Member.class, id);         // Member일 것 같지만 Proxy!
System.out.println(ref.getClass() == find.getClass()); // true
```

두 경우 모두 이유는 같다: **JPA는 같은 트랜잭션(영속성 컨텍스트) 안에서 같은 엔티티의 `==` 동일성을 보장**해야 하므로, 먼저 들어온 쪽의 타입으로 맞춰준다. 결론 — 프록시든 아니든 신경 쓰지 말고 `instanceof`로 비교하면 된다.

### 10-3. 준영속 상태에서 프록시 초기화 (`85aee62`)

```java
Member refMember = em.getReference(Member.class, member1.getId());

em.detach(refMember); // 또는 em.clear()
refMember.getUsername();
// org.hibernate.LazyInitializationException: could not initialize proxy - no Session
```

⚠️ **최신 하이버네이트에서 `em.close()`로는 예외가 재현되지 않는다.** 트랜잭션이 진행 중이면 실제 자원 해제가 커밋/롤백 시점까지 미뤄져서, 그 사이의 프록시 초기화는 성공한다. 예외 재현은 `detach(entity)` 또는 `clear()`로 한다. (예외 메시지도 다르다 — detach/clear: `no Session`, close: `the owning Session was closed`)

### 10-4. 프록시 확인·강제 초기화 유틸 (`1a7a1a7`)

```java
Member refMember = em.getReference(Member.class, member1.getId());
System.out.println(refMember.getClass()); // 프록시 클래스 확인

emf.getPersistenceUnitUtil().isLoaded(refMember); // false
Hibernate.initialize(refMember); // 강제 초기화 (Hibernate 전용, JPA 표준엔 없음)
emf.getPersistenceUnitUtil().isLoaded(refMember); // true
```

### 10-5. `FetchType.LAZY` 적용 (`69d6416`)

```java
// Member.java
@ManyToOne(fetch = FetchType.LAZY) // LAZY : Member만 db 조회, team에는 프록시가 들어감
@JoinColumn(name = "TEAM_ID")
private Team team;
```

```java
Member m = em.find(Member.class, member1.getId()); // Member만 SELECT
m.getTeam().getClass();  // Team 프록시
m.getTeam().getName();   // 이 시점에 Team SELECT (프록시 초기화)
```

### 10-6. JPQL N+1 문제 — 왜 전부 LAZY여야 하는가 (`6c4e769`)

teamA/member1, teamB/member2를 저장해두고 EAGER 상태에서 JPQL을 실행하면:

```java
List<Member> members = em.createQuery("select m from Member m", Member.class).getResultList();
// SQL: Member 전체 SELECT 1번 + 각 회원의 Team SELECT N번 = N+1 문제
```

JPQL은 SQL로 그대로 번역돼 Member만 가져오는데, EAGER가 "즉시 채워라"를 강제하니 팀 수만큼 추가 쿼리가 나간다. 해결은 **전부 LAZY로 두고 필요한 곳에서 fetch join**:

```java
List<Member> members = em.createQuery(
        "select m from Member m join fetch m.team", Member.class).getResultList();
// 조인 쿼리 1번으로 Member + Team 함께 조회
```

### 10-7. 영속성 전이 + 고아 객체 (`a047ef9`, `79af8cf`)

```java
// Parent.java — 단일 소유자(부모) + 라이프사이클 동일할 때만 사용
@OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Child> childList = new ArrayList<>();
```

```java
parent.addChild(child1);
parent.addChild(child2);
em.persist(parent); // cascade 덕에 child persist 생략 — INSERT 3번 나감

Parent findParent = em.find(Parent.class, parent.getId());

findParent.getChildList().remove(0); // orphanRemoval = true → 커밋 시 DELETE
em.remove(findParent);               // cascade REMOVE → 부모+자식 전부 DELETE
```

⚠️ 실습에서 확인한 함정: **`cascade = ALL`만으로는 컬렉션에서 뺀 자식이 삭제되지 않는다.** cascade의 REMOVE 전이는 `em.remove(parent)`에만 반응하고, 컬렉션 제거에 반응하는 건 `orphanRemoval = true`다. (`mappedBy` 쪽에서 빼기만 하면 UPDATE조차 안 나간다.) 반대로 부모 자체를 `em.remove()`할 때는 orphanRemoval 없이 cascade만으로도 자식까지 지워진다.

## 10. 강의(구버전)와 내 코드의 차이

| 항목 | 강의 자료 | 내 코드 |
|------|-----------|---------|
| 패키지 | `javax.persistence` | **`jakarta.persistence`** (Hibernate 6.x) |
| 프록시 클래스명 | `...javassist...` 로 소개 | Hibernate 6는 ByteBuddy 기반 `...HibernateProxy...` |
| 프록시 예제 | `Member` ↔ `Team` (hellojpa) | 05장 코드를 `hellojpa`로 복원해 동일하게 실습 (9절) |
| `em.close()` 후 프록시 초기화 | 즉시 `LazyInitializationException` | 트랜잭션 종료까지 자원 해제가 지연돼 **예외 안 남** → `detach`/`clear`로 재현 |
| fetch / cascade | 실전 예제 5에서 LAZY + ALL 적용 | ✅ 적용 완료 (`0ea92ab`) |

## ✅ 핵심 요약

1. `em.find()`는 즉시 조회, `em.getReference()`는 **프록시**를 반환해 실제 사용 시점까지 DB 조회를 미룬다.
2. 프록시는 원본을 상속한 가짜 객체로, 초기화돼도 실제 엔티티로 **바뀌지 않고** target에 위임한다.
3. 타입 비교는 `instanceof`, 준영속 상태 초기화는 `LazyInitializationException` 주의 (최신 하이버네이트는 `em.close()`로는 재현 안 됨 → `detach`/`clear` 사용).
4. `@ManyToOne`/`@OneToOne`의 기본값은 **EAGER** → **모든 연관관계는 LAZY로 명시**하는 게 실무 표준.
5. 즉시 로딩은 예상 못 한 조인과 **JPQL N+1 문제**의 주범. 함께 조회는 fetch join으로 해결.
6. `cascade`는 연관관계 매핑이 아니라 **영속성 상태 전파** 편의 기능 — 단일 소유 관계에서만 사용.
7. `orphanRemoval = true`는 컬렉션에서 빠진 자식까지 DELETE — `CascadeType.ALL`과 조합하면 부모가 자식 **생명주기를 완전히 관리** (Aggregate Root).
8. 실전 예제 5: 전부 LAZY + `Order → Delivery/OrderItem`에 `CascadeType.ALL`.
