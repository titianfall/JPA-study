## 1. JPA가 지원하는 다양한 쿼리 방법

| 방법 | 정체 | 실무 평가 |
|------|------|-----------|
| **JPQL** | SQL을 추상화한 **객체 지향 쿼리 언어** (표준) | 기본기, 반드시 숙지 |
| **Criteria** | 자바 코드로 JPQL을 조립하는 빌더 (표준) | 너무 복잡, 실용성 없음 → **QueryDSL 권장** |
| **QueryDSL** | 자바 코드로 JPQL을 조립하는 빌더 (오픈소스) | 컴파일 시점 문법 오류 검출, 동적 쿼리 편리 → **실무 권장** |
| **네이티브 SQL** | SQL을 직접 사용 | DB 종속 기능 필요할 때 (예: 오라클 `CONNECT BY`) |
| **JDBC API 직접, MyBatis, JdbcTemplate** | JPA와 함께 사용 가능 | ⚠️ 영속성 컨텍스트를 적절한 시점에 **수동 플러시** 필요 |

### 왜 JPQL이 필요한가?
- 가장 단순한 조회는 `em.find()`, 연관은 객체 그래프 탐색(`a.getB().getC()`)으로 충분.
- **문제는 검색 쿼리.** "나이 18살 이상 회원 전체" 같은 조건 검색은
  모든 DB 데이터를 객체로 변환해서 거를 수 없다 → **검색 조건이 포함된 쿼리가 필요**.
- JPQL은 **테이블이 아닌 엔티티 객체를 대상**으로 쿼리하고, SQL로 번역되어 실행된다.

```java
String jpql = "select m from Member m where m.age > 18";
List<Member> result = em.createQuery(jpql, Member.class).getResultList();
```

```sql
-- 실행된 SQL
select m.id, m.age, m.USERNAME, m.TEAM_ID
from Member m
where m.age > 18
```

## 2. JPQL 기본 문법

```
select_문 ::= select절 from절 [where절] [groupby절] [having절] [orderby절]
update_문 ::= update절 [where절]
delete_문 ::= delete절 [where절]
```

- `select m from Member as m where m.age > 18`
- **엔티티와 속성은 대소문자 구분 O** (`Member`, `age`)
- **JPQL 키워드는 대소문자 구분 X** (`SELECT`, `from`, `where`)
- **테이블 이름이 아닌 엔티티 이름** 사용 (`Member`)
- **별칭(m)은 필수**, `as`는 생략 가능
- 집합 함수: `COUNT(m)`, `SUM(m.age)`, `AVG(m.age)`, `MAX(m.age)`, `MIN(m.age)`
- `GROUP BY`, `HAVING`, `ORDER BY` 모두 SQL과 동일하게 지원

### TypedQuery vs Query

```java
// 반환 타입이 명확할 때 → TypedQuery
TypedQuery<Member> query =
        em.createQuery("SELECT m FROM Member m", Member.class);

// 반환 타입이 명확하지 않을 때 (username은 String, age는 int) → Query
Query query =
        em.createQuery("SELECT m.username, m.age from Member m");
```

### 결과 조회 API

| API | 반환 | 결과 없음 | 둘 이상 |
|-----|------|----------|---------|
| `getResultList()` | 리스트 | **빈 리스트** (안전) | 그대로 반환 |
| `getSingleResult()` | 단일 객체 | ⚠️ `NoResultException` | ⚠️ `NonUniqueResultException` |

> ⚠️ 강의(구버전)는 `javax.persistence.NoResultException`이지만
> 내 프로젝트(Hibernate 6.x)는 **`jakarta.persistence.NoResultException`** 이다.

### 파라미터 바인딩

```java
// 이름 기준 (권장)
em.createQuery("SELECT m FROM Member m where m.username = :username", Member.class)
        .setParameter("username", usernameParam);

// 위치 기준 (순서가 밀리면 장애로 이어짐 → 비권장)
em.createQuery("SELECT m FROM Member m where m.username = ?1", Member.class)
        .setParameter(1, usernameParam);
```

## 3. 프로젝션 — SELECT 절에 조회할 대상 지정

| JPQL | 종류 |
|------|------|
| `SELECT m FROM Member m` | 엔티티 프로젝션 |
| `SELECT m.team FROM Member m` | 엔티티 프로젝션 (⚠️ 묵시적 조인 발생) |
| `SELECT m.address FROM Member m` | 임베디드 타입 프로젝션 |
| `SELECT m.username, m.age FROM Member m` | 스칼라 타입 프로젝션 |

- `DISTINCT`로 중복 제거 가능.

### 여러 값 조회 — 3가지 방법
1. `Query` 타입으로 조회
2. `Object[]` 타입으로 조회
3. **`new` 명령어로 DTO 바로 조회 (실무 권장)**

```java
SELECT new jpabook.jpql.UserDTO(m.username, m.age) FROM Member m
```
- 패키지 명을 **포함한 전체 클래스 명** 입력
- **순서와 타입이 일치하는 생성자** 필요

## 4. 페이징 API

JPA는 페이징을 딱 두 API로 추상화한다.
- `setFirstResult(int startPosition)` : 조회 시작 위치 (**0부터**)
- `setMaxResults(int maxResult)` : 조회할 데이터 수

내 코드(`study/hello-jpql`에서 실습):

```java
List<Member> result = em.createQuery("select m from Member as m", Member.class)
        .setFirstResult(5)   // 6번째부터 (0-based, 앞의 5개 건너뜀)
        .setMaxResults(8)    // 최대 8개
        .getResultList();
```

- 방언(Dialect)이 DB에 맞는 SQL로 번역: H2/MySQL은 `LIMIT ?, ?`, Oracle은 `ROWNUM` 3중 서브쿼리
- **JPQL은 그대로 두고 DB만 바꿔도 페이징이 동작** → 방언의 힘

## 5. 조인

```sql
-- 내부 조인
SELECT m FROM Member m [INNER] JOIN m.team t
-- 외부 조인
SELECT m FROM Member m LEFT [OUTER] JOIN m.team t
-- 세타 조인 (연관관계 없는 엔티티끼리)
select count(m) from Member m, Team t where m.username = t.name
```

### ON 절 (JPA 2.1+)

```sql
-- 1. 조인 대상 필터링: 팀 이름이 'A'인 팀만 조인
SELECT m, t FROM Member m LEFT JOIN m.team t on t.name = 'A'

-- 2. 연관관계 없는 엔티티 외부 조인 (하이버네이트 5.1+)
SELECT m, t FROM Member m LEFT JOIN Team t on m.username = t.name
```

## 6. 서브 쿼리

```sql
-- 나이가 평균보다 많은 회원
select m from Member m
where m.age > (select avg(m2.age) from Member m2)

-- 한 건이라도 주문한 고객
select m from Member m
where (select count(o) from Order o where m = o.member) > 0
```

### 지원 함수
- `[NOT] EXISTS (subquery)` : 결과가 존재하면 참
- `{ALL | ANY | SOME} (subquery)` : ALL은 모두 만족, ANY/SOME은 하나라도 만족
- `[NOT] IN (subquery)` : 결과 중 하나라도 같으면 참

### 한계
- JPA 표준은 **WHERE, HAVING 절에서만** 서브 쿼리 가능 (SELECT 절은 하이버네이트가 지원)
- ⚠️ 강의(구버전)에서는 FROM 절 서브쿼리 불가라고 하지만,
  **하이버네이트 6부터는 FROM 절 서브쿼리를 지원**한다.
  그래도 조인으로 풀 수 있으면 조인으로 해결하는 것이 우선.

## 7. 타입 표현 · 조건식 · 함수

### JPQL 타입 표현
| 종류 | 예시 |
|------|------|
| 문자 | `'HELLO'`, `'She''s'` |
| 숫자 | `10L`(Long), `10D`(Double), `10F`(Float) |
| Boolean | `TRUE`, `FALSE` |
| ENUM | `jpabook.MemberType.Admin` (**패키지명 포함**) |
| 엔티티 타입 | `TYPE(m) = Member` (상속 관계에서 사용) |

- SQL과 같은 식 그대로 지원: `EXISTS`, `IN`, `AND/OR/NOT`, `=, >, >=, <, <=, <>`, `BETWEEN`, `LIKE`, `IS NULL`

### CASE 식

```sql
-- 기본 CASE
select case when m.age <= 10 then '학생요금'
            when m.age >= 60 then '경로요금'
            else '일반요금' end
from Member m

-- 단순 CASE
select case t.name when '팀A' then '인센티브110%'
                   else '인센티브105%' end
from Team t
```

```sql
-- COALESCE: 하나씩 조회해서 null이 아니면 반환
select coalesce(m.username, '이름 없는 회원') from Member m

-- NULLIF: 두 값이 같으면 null, 다르면 첫 번째 값
select NULLIF(m.username, '관리자') from Member m
```

### 함수
- **JPQL 기본 함수**: `CONCAT`, `SUBSTRING`, `TRIM`, `LOWER/UPPER`, `LENGTH`, `LOCATE`, `ABS/SQRT/MOD`, `SIZE/INDEX`(JPA 용도)
- **문자열 연결**: 표준 함수는 `concat('a', 'b')`처럼 인자 2개가 기본이다. Hibernate/H2에서는 `'||'`도 SQL로 번역되지만, JPQL 반환값은 `String`이므로 `createQuery(jpql, String.class)`로 받아야 한다.
- **`TRIM`, `LOWER`, `UPPER`**: 문자열 정규화 조건에 사용한다.
- **`LOCATE`**: 부분 문자열 위치를 반환한다. 예: `locate('de', 'abcdef')`
- **`ABS`, `MOD`**: 숫자 계산에 사용한다. 예: `abs(m.age - 30) < 5`, `mod(m.age, 2) = 0`
- **`SIZE`**: 컬렉션 크기 조회. 예: `size(t.members)`
- **`INDEX`**: `@OrderColumn`이 있는 `List` 컬렉션에서 위치값 조회. 현재 실습의 `Team.members`에는 `@OrderColumn`이 없으므로 일반 컬렉션처럼 쓰면 안 된다.
- **사용자 정의 함수**: 사용하는 DB 방언을 상속받아 등록 후
  `select function('group_concat', m.username) from Member m`

### Hibernate 6에서 사용자 정의 함수 등록

강의의 구버전 예제처럼 생성자에서 `registerFunction(...)`을 호출하면 Hibernate 6.4 기준으로는 메서드를 찾을 수 없다.
현재 실습 코드는 `H2Dialect`를 상속하고 `initializeFunctionRegistry(...)`를 오버라이드해서 등록한다.

```java
public class MyH2Dialect extends H2Dialect {

    @Override
    public void initializeFunctionRegistry(FunctionContributions functionContributions) {
        super.initializeFunctionRegistry(functionContributions);

        functionContributions.getFunctionRegistry().register(
                "group_concat",
                new StandardSQLFunction("group_concat", StandardBasicTypes.STRING)
        );
    }
}
```

`function('group_concat', m.username)`을 쓰려면 `persistence.xml`에서 Hibernate 기본 `H2Dialect` 대신 사용자 Dialect를 가리켜야 한다.
실습 패키지명이 `Dialect`라면 값은 대소문자까지 맞춰 `Dialect.MyH2Dialect`가 되어야 한다.

## 8. 경로 표현식 — `.`(점)으로 객체 그래프 탐색

```sql
select m.username    -- 상태 필드
from Member m
  join m.team t      -- 단일 값 연관 필드
  join t.members tm  -- 컬렉션 값 연관 필드
where t.name = '팀A'
```

| 종류 | 정의 | 묵시적 조인 | 추가 탐색 |
|------|------|------------|----------|
| **상태 필드** | 값을 저장하는 필드 (`m.username`) | X | X (탐색의 끝) |
| **단일 값 연관 필드** | `@ManyToOne`, `@OneToOne` (`m.team`) | ⚠️ **묵시적 내부 조인 발생** | O |
| **컬렉션 값 연관 필드** | `@OneToMany`, `@ManyToMany` (`t.members`) | ⚠️ 묵시적 내부 조인 발생 | X |

```sql
select o.member.team from Order o            -- 성공 (조인 2번 발생!)
select t.members from Team t                 -- 성공
select t.members.username from Team t        -- ❌ 실패 (컬렉션에서 추가 탐색 불가)
select m.username from Team t join t.members m  -- 성공 (명시적 조인으로 별칭 획득)
```

> 현재 실습 엔티티 기준으로 `Member`에는 `orders` 컬렉션이 없다.
> 그래서 `size(m.orders)` 같은 쿼리는 JPQL 문제가 아니라 매핑 자체가 없어서 IDE/런타임에서 실패한다.
> 컬렉션 함수 예시는 실제 매핑이 있는 `Team.members` 기준으로 생각하면 된다.

### 💡 실무 조언
- **묵시적 조인 대신 명시적 조인(`join` 키워드) 사용.**
- 조인은 SQL 튜닝의 중요 포인트인데, 묵시적 조인은 조인이 일어나는 상황을 한눈에 파악하기 어렵다.
- 묵시적 조인은 항상 **내부 조인만** 가능하고, SELECT/WHERE 절에서 썼는데 SQL의 FROM 절에 영향을 준다.

## 9. 페치 조인 (fetch join) — ⭐ 실무에서 정말정말 중요

- **SQL의 조인 종류가 아니다.** JPQL이 **성능 최적화**를 위해 제공하는 기능.
- 연관된 엔티티나 컬렉션을 **SQL 한 번에 함께 조회**.
- 문법: `[ LEFT [OUTER] | INNER ] JOIN FETCH 조인경로`

### 엔티티 페치 조인 (N:1)

```java
String jpql = "select m from Member m join fetch m.team";
List<Member> members = em.createQuery(jpql, Member.class).getResultList();

for (Member member : members) {
    // 팀을 이미 함께 조회했으므로 지연 로딩(추가 SQL) 발생 X
    System.out.println(member.getUsername() + ", " + member.getTeam().getName());
}
```

```sql
SELECT M.*, T.* FROM MEMBER M
INNER JOIN TEAM T ON M.TEAM_ID = T.ID
```

> 💡 페치 조인이 없으면? 지연 로딩 상태에서 루프를 돌 때마다 팀 조회 SQL이 나간다
> → **N+1 문제**. 페치 조인이 그 해결책.

### 컬렉션 페치 조인 (1:N)

```sql
select t from Team t join fetch t.members where t.name = '팀A'
```

- 1:N 조인은 DB row가 뻥튀기된다(팀A에 회원 2명 → 팀A가 2줄).

### 페치 조인과 DISTINCT

- JPQL의 `DISTINCT`는 2가지 기능: ① SQL에 DISTINCT 추가 ② **애플리케이션에서 엔티티 중복 제거**
- SQL DISTINCT만으로는 row 데이터가 달라 중복 제거 실패 →
  같은 식별자를 가진 `Team` 엔티티를 애플리케이션에서 제거.

> ⚠️ 강의(구버전)와 차이 — **하이버네이트 6부터는 DISTINCT를 쓰지 않아도**
> 컬렉션 페치 조인 시 애플리케이션 레벨 중복 제거가 **자동 적용**된다.
> (내 프로젝트는 Hibernate 6.x이므로 자동 적용 대상)

### 페치 조인 vs 일반 조인

| | 일반 조인 | 페치 조인 |
|---|----------|----------|
| SQL | 조인은 하지만 **SELECT 절에는 대상 엔티티만** (`SELECT T.*`) | 연관 엔티티까지 함께 (`SELECT T.*, M.*`) |
| 연관 엔티티 | 조회 X → 이후 접근 시 지연 로딩 | **함께 조회 (즉시 로딩)** |

- JPQL은 결과를 반환할 때 연관관계를 고려하지 않는다. 페치 조인만이 객체 그래프를 SQL 한 번에 조회하는 개념.

### 페치 조인의 한계

- **페치 조인 대상에는 별칭을 줄 수 없다** (하이버네이트는 가능하지만 가급적 사용 X).
- **둘 이상의 컬렉션**은 페치 조인 불가.
- **컬렉션을 페치 조인하면 페이징 API 사용 불가.**
  - 일대일·다대일 같은 **단일 값 연관 필드는 페치 조인해도 페이징 가능**.
  - ⚠️ 하이버네이트는 경고 로그만 남기고 **메모리에서 페이징** → 매우 위험.
- 엔티티에 직접 적용하는 글로벌 로딩 전략(`@OneToMany(fetch = LAZY)`)보다 **우선**한다.

### 💡 실무 전략
1. 글로벌 로딩 전략은 **모두 지연 로딩(LAZY)**.
2. 최적화가 필요한 곳에만 **페치 조인** 적용.
3. 엔티티 모양이 아닌 전혀 다른 결과가 필요하면 → 일반 조인 + **필요한 데이터만 DTO로 조회**가 효과적.

## 10. 다형성 쿼리 (상속 관계)

```sql
-- TYPE: 조회 대상을 특정 자식으로 한정
select i from Item i where type(i) IN (Book, Movie)
-- SQL: where i.DTYPE in ('B', 'M')

-- TREAT (JPA 2.1): 자바의 타입 캐스팅처럼 부모를 자식 타입으로 다룸
select i from Item i where treat(i as Book).author = 'kim'
-- SQL: where i.DTYPE = 'B' and i.author = 'kim'
```

## 11. 엔티티 직접 사용

- JPQL에서 엔티티를 직접 사용하면 SQL에서는 그 엔티티의 **기본 키 값**으로 치환된다.

```java
select count(m) from Member m       // 엔티티 직접 사용
select count(m.id) from Member m    // 아이디 사용
// → 둘 다 같은 SQL: select count(m.id) as cnt from Member m
```

```java
// 엔티티를 파라미터로 전달해도, 식별자를 전달해도 같은 SQL
em.createQuery("select m from Member m where m = :member")
        .setParameter("member", member);
// SQL: select m.* from Member m where m.id = ?

// 외래 키 값도 동일
em.createQuery("select m from Member m where m.team = :team")
        .setParameter("team", team);
// SQL: select m.* from Member m where m.team_id = ?
```

## 12. Named 쿼리 — 정적 쿼리

- 미리 정의해서 **이름을 부여**해두고 사용하는 JPQL. 어노테이션 또는 XML에 정의.
- **애플리케이션 로딩 시점에 초기화 후 재사용** (SQL로 미리 파싱해 캐시).
- ✅ 가장 큰 장점: **애플리케이션 로딩 시점에 쿼리 문법을 검증** → 오타가 배포 전에 잡힌다.

```java
@Entity
@NamedQuery(
        name = "Member.findByUsername",
        query = "select m from Member m where m.username = :username")
public class Member { ... }

List<Member> resultList =
        em.createNamedQuery("Member.findByUsername", Member.class)
                .setParameter("username", "회원1")
                .getResultList();
```

- XML(`META-INF/ormMember.xml`)에 정의하면 **XML이 항상 우선권**. 운영 환경별로 다른 XML 배포 가능.
- 💡 Spring Data JPA에서는 `@Query`가 이 Named 쿼리를 이름 없이 쓰는 셈.

## 13. 벌크 연산

- "재고 10개 미만 상품 가격 10% 인상" — 변경 감지로 처리하면 **100건이면 UPDATE SQL 100번**.
- 벌크 연산은 **쿼리 한 번으로 여러 테이블 로우 변경**.

```java
int resultCount = em.createQuery(
        "update Product p set p.price = p.price * 1.1 " +
        "where p.stockAmount < :stockAmount")
        .setParameter("stockAmount", 10)
        .executeUpdate();   // 영향받은 엔티티 수 반환
```

- `UPDATE`, `DELETE` 지원 (+`INSERT INTO ... SELECT`는 하이버네이트 지원)

### ⚠️ 벌크 연산 주의
- 벌크 연산은 **영속성 컨텍스트를 무시하고 DB에 직접 쿼리**한다.
  → 영속성 컨텍스트의 엔티티와 DB 상태가 어긋날 수 있다.
- 해결책 두 가지:
  1. **벌크 연산을 가장 먼저 실행**
  2. **벌크 연산 수행 후 영속성 컨텍스트 초기화** (`em.clear()`)

---

## ✅ 핵심 요약 한 줄 정리
1. **JPQL** = 엔티티 객체 대상 쿼리, SQL로 번역됨. 동적 쿼리는 **QueryDSL**(Criteria는 비권장).
2. 별칭 필수, 파라미터는 **이름 기준** 바인딩, DTO 조회는 **`new` 명령어**.
3. 페이징은 `setFirstResult` / `setMaxResults` 두 개로 끝 — 방언이 DB별 SQL 처리.
4. **묵시적 조인 금지, 명시적 조인 사용** — 컬렉션 경로는 탐색의 끝.
5. **페치 조인** = N+1 해결의 핵심. 단, 컬렉션 페치 조인 + 페이징 금지, 둘 이상 컬렉션 금지.
6. 하이버네이트 6 변경: **DISTINCT 자동 중복 제거**, **FROM 절 서브쿼리 지원**.
7. **Named 쿼리**는 로딩 시점에 문법 검증되는 정적 쿼리.
8. **벌크 연산**은 영속성 컨텍스트를 무시 → 먼저 실행하거나 실행 후 `em.clear()`.
