## 8. 경로 표현식 — `.`(점)으로 객체 그래프 탐색

```sql
select m.username    -- 상태 필드
from Member m
  join m.team t      -- 단일 값 연관 필드
  join t.members tm  -- 컬렉션 값 연관 필드
where t.name = '팀A'
```

| 종류 | 정의 | 묵시적 조인 | 추가 탐색 |
|------|------|------------|----------|
| **상태 필드** | 값을 저장하는 필드 (`m.username`) | X | X (탐색의 끝) |
| **단일 값 연관 필드** | `@ManyToOne`, `@OneToOne` (`m.team`) | ⚠️ **묵시적 내부 조인 발생** | O |
| **컬렉션 값 연관 필드** | `@OneToMany`, `@ManyToMany` (`t.members`) | ⚠️ 묵시적 내부 조인 발생 | X |

```sql
select o.member.team from Order o            -- 성공 (조인 2번 발생!)
select t.members from Team t                 -- 성공
select t.members.username from Team t        -- ❌ 실패 (컬렉션에서 추가 탐색 불가)
select m.username from Team t join t.members m  -- 성공 (명시적 조인으로 별칭 획득)
```

> 현재 실습 엔티티 기준으로 `Member`에는 `orders` 컬렉션이 없다.
> 그래서 `size(m.orders)` 같은 쿼리는 JPQL 문제가 아니라 매핑 자체가 없어서 IDE/런타임에서 실패한다.
> 컬렉션 함수 예시는 실제 매핑이 있는 `Team.members` 기준으로 생각하면 된다.

### 💡 실무 조언
- **묵시적 조인 대신 명시적 조인(`join` 키워드) 사용.**
- 조인은 SQL 튜닝의 중요 포인트인데, 묵시적 조인은 조인이 일어나는 상황을 한눈에 파악하기 어렵다.
- 묵시적 조인은 항상 **내부 조인만** 가능하고, SELECT/WHERE 절에서 썼는데 SQL의 FROM 절에 영향을 준다.

## 9. 페치 조인 (fetch join) — ⭐ 실무에서 정말정말 중요

- **SQL의 조인 종류가 아니다.** JPQL이 **성능 최적화**를 위해 제공하는 기능.
- 연관된 엔티티나 컬렉션을 **SQL 한 번에 함께 조회**.
- 문법: `[ LEFT [OUTER] | INNER ] JOIN FETCH 조인경로`

### 엔티티 페치 조인 (N:1)

```java
String jpql = "select m from Member m join fetch m.team";
List<Member> members = em.createQuery(jpql, Member.class).getResultList();

for (Member member : members) {
    // 팀을 이미 함께 조회했으므로 지연 로딩(추가 SQL) 발생 X
    System.out.println(member.getUsername() + ", " + member.getTeam().getName());
}
```

```sql
SELECT M.*, T.* FROM MEMBER M
INNER JOIN TEAM T ON M.TEAM_ID = T.ID
```

> 💡 페치 조인이 없으면? 지연 로딩 상태에서 루프를 돌 때마다 팀 조회 SQL이 나간다
> → **N+1 문제**. 페치 조인이 그 해결책.

### 컬렉션 페치 조인 (1:N)

```sql
select t from Team t join fetch t.members where t.name = '팀A'
```

- 1:N 조인은 DB row가 뻥튀기된다(팀A에 회원 2명 → 팀A가 2줄).

### 페치 조인과 DISTINCT

- JPQL의 `DISTINCT`는 2가지 기능: ① SQL에 DISTINCT 추가 ② **애플리케이션에서 엔티티 중복 제거**
- SQL DISTINCT만으로는 row 데이터가 달라 중복 제거 실패 →
  같은 식별자를 가진 `Team` 엔티티를 애플리케이션에서 제거.

> ⚠️ 강의(구버전)와 차이 — **하이버네이트 6부터는 DISTINCT를 쓰지 않아도**
> 컬렉션 페치 조인 시 애플리케이션 레벨 중복 제거가 **자동 적용**된다.
> (내 프로젝트는 Hibernate 6.x이므로 자동 적용 대상)

### 페치 조인 vs 일반 조인

| | 일반 조인 | 페치 조인 |
|---|----------|----------|
| SQL | 조인은 하지만 **SELECT 절에는 대상 엔티티만** (`SELECT T.*`) | 연관 엔티티까지 함께 (`SELECT T.*, M.*`) |
| 연관 엔티티 | 조회 X → 이후 접근 시 지연 로딩 | **함께 조회 (즉시 로딩)** |

- JPQL은 결과를 반환할 때 연관관계를 고려하지 않는다. 페치 조인만이 객체 그래프를 SQL 한 번에 조회하는 개념.

### 페치 조인의 한계

- **페치 조인 대상에는 별칭을 줄 수 없다** (하이버네이트는 가능하지만 가급적 사용 X).
  - 이유: JPA의 객체 그래프는 `team.getMembers()` 호출 시 **회원 "전부"가 나온다는 전제** 위에 서 있다.
    별칭으로 걸러서(`join fetch t.members m where m.age > 10`) 가져오면 컬렉션이 일부만 담긴 채
    영속성 컨텍스트에 올라가고, cascade·변경 감지가 그 불완전한 상태 기준으로 동작한다.
  - 걸러진 회원이 필요하면 그래프 탐색 말고 **Member 를 대상으로 따로 쿼리**할 것.
- **둘 이상의 컬렉션**은 페치 조인 불가.
- **컬렉션을 페치 조인하면 페이징 API 사용 불가.**
  - 일대일·다대일 같은 **단일 값 연관 필드는 페치 조인해도 페이징 가능**.
  - ⚠️ 하이버네이트는 경고 로그만 남기고 **메모리에서 페이징** → 매우 위험.
- 엔티티에 직접 적용하는 글로벌 로딩 전략(`@OneToMany(fetch = LAZY)`)보다 **우선**한다.

### 컬렉션 + 페이징의 대안 — `@BatchSize`

페치 조인 대신 지연 로딩을 유지하면서, 컬렉션을 **IN 쿼리로 묶어서** 가져온다.

```java
@OneToMany(fetch = FetchType.LAZY, mappedBy = "team")
@BatchSize(size = 100)   // org.hibernate.annotations.BatchSize (컬렉션 필드에!)
private List<Member> members = new ArrayList<>();
```

```sql
-- N+1 이 아니라: 팀 조회 1번 + IN 쿼리 1번
select ... from Member m1_0 where m1_0.TEAM_ID in (?, ?, ?, ..., ?)
```

- ⚠️ 위치 주의 — 클래스에 붙이면 "엔티티 프록시 배치"가 되어 **컬렉션 로딩에는 적용 안 됨**.
  컬렉션을 배치하려면 **컬렉션 필드**에 붙여야 한다. (직접 틀려봄: `Member` 클래스에 붙였더니 여전히 `TEAM_ID=?` 단건 N방)
- `hibernate.jdbc.batch_size`(쓰기 배치)와는 **전혀 다른 설정**. 조회 글로벌 설정은 `hibernate.default_batch_fetch_size`.
- ⚠️ Hibernate 6 차이 — 팀이 2개여도 IN 절 물음표가 **size 만큼(100개) 패딩**되어 나간다.
  남는 칸은 마지막 id 반복 바인딩. SQL 모양을 고정해 실행 계획/statement 캐시를 재사용하기 위함
  (Hibernate 5 는 100 → 50 → 25 ... 하위 배치로 쪼개는 방식이었다).

### 💡 실무 전략
1. 글로벌 로딩 전략은 **모두 지연 로딩(LAZY)**.
2. 최적화가 필요한 곳에만 **페치 조인** 적용.
3. 컬렉션 + 페이징이 필요하면 → 지연 로딩 + `@BatchSize`(또는 `default_batch_fetch_size`).
4. 엔티티 모양이 아닌 전혀 다른 결과가 필요하면 → 일반 조인 + **필요한 데이터만 DTO로 조회**가 효과적.

## 10. 다형성 쿼리 (상속 관계)

```sql
-- TYPE: 조회 대상을 특정 자식으로 한정
select i from Item i where type(i) IN (Book, Movie)
-- SQL: where i.DTYPE in ('B', 'M')

-- TREAT (JPA 2.1): 자바의 타입 캐스팅처럼 부모를 자식 타입으로 다룸
select i from Item i where treat(i as Book).author = 'kim'
-- SQL: where i.DTYPE = 'B' and i.author = 'kim'
```

## 11. 엔티티 직접 사용

- JPQL에서 엔티티를 직접 사용하면 SQL에서는 그 엔티티의 **기본 키 값**으로 치환된다.

```java
select count(m) from Member m       // 엔티티 직접 사용
select count(m.id) from Member m    // 아이디 사용
// → 둘 다 같은 SQL: select count(m.id) as cnt from Member m
```

```java
// 엔티티를 파라미터로 전달해도, 식별자를 전달해도 같은 SQL
em.createQuery("select m from Member m where m = :member")
        .setParameter("member", member);          // Member 객체
em.createQuery("select m from Member m where m.id = :memberId")
        .setParameter("memberId", member.getId()); // Long 값
// SQL: 둘 다 select m.* from Member m where m.MEMBER_ID = ?

// 외래 키 값도 동일
em.createQuery("select m from Member m where m.team = :team")
        .setParameter("team", team);
// SQL: select m.* from Member m where m.TEAM_ID = ?
```

> ⚠️ 두 형태를 섞으면 안 된다. `where m = :param` 에 `getId()`(Long)를 바인딩하면
> `QueryArgumentException: Argument of type [Long] did not match parameter type [Member]` —
> JPQL 은 타입을 검사하므로 **엔티티 비교엔 엔티티를, id 비교(`m.id`)엔 식별자 값을** 넣어야 한다.

## 12. Named 쿼리 — 정적 쿼리

- 미리 정의해서 **이름을 부여**해두고 사용하는 JPQL. 어노테이션 또는 XML에 정의.
- **애플리케이션 로딩 시점에 초기화 후 재사용** (SQL로 미리 파싱해 캐시).
- ✅ 가장 큰 장점: **애플리케이션 로딩 시점에 쿼리 문법을 검증** → 오타가 배포 전에 잡힌다.

```java
@Entity
@NamedQuery(
        name = "Member.findByUsername",
        query = "select m from Member m where m.username = :username")
public class Member { ... }

List<Member> resultList =
        em.createNamedQuery("Member.findByUsername", Member.class)
                .setParameter("username", "회원1")
                .getResultList();
```

- XML(`META-INF/ormMember.xml`)에 정의하면 **XML이 항상 우선권**. 운영 환경별로 다른 XML 배포 가능.
- ⚠️ `persistence.xml` 에 등록할 때 `<mapping-file>` 은 `<properties>` **안이 아니라 밖**,
  그리고 `<properties>` **보다 앞**에 와야 한다 (persistence-unit 자식 요소는 스키마상 순서 고정).

  ```xml
  <persistence-unit name="hellojpql">
      <mapping-file>META-INF/ormMember.xml</mapping-file>  <!-- properties 보다 먼저 -->
      <properties> ... </properties>
  </persistence-unit>
  ```
- 💡 Spring Data JPA에서는 `@Query`가 이 Named 쿼리를 이름 없이 쓰는 셈.

## 12.5 JPQL 과 영속성 컨텍스트

**JPQL 은 1차 캐시가 있어도 항상 SQL 을 DB 로 보낸다.** "캐시에 있으면 쿼리 생략"은 `em.find()`(기본 키 단건 조회) 전용 최적화다.

| 조회 방식 | SQL 생략 가능? | 1차 캐시 역할 |
|-----------|---------------|--------------|
| `em.find()` | ✅ 캐시에 있으면 생략 | 조회 전 먼저 확인 |
| JPQL 엔티티 조회 | ❌ 항상 SQL 발사 | **결과 조립 시** 같은 식별자면 기존 엔티티로 대체 (동일성 보장) |
| JPQL 스칼라 조회 | ❌ 항상 SQL 발사 | 관여 안 함 (엔티티 조립 단계가 없음) |

- JPQL 은 어떤 조건이 올지 모르므로 1차 캐시(id → 엔티티 맵)로는 답을 만들 수 없다. DB 만이 전체를 안다.
- 쿼리는 나가되, row 의 식별자가 이미 1차 캐시에 있으면 **DB 에서 읽은 값은 버리고 캐시의 기존 엔티티를 반환**.
- 같은 이유로 JPQL 실행 직전에 자동 flush 가 일어난다 (DB 기준으로 답을 만들기 위해 싱크를 맞춤).

## 13. 벌크 연산

- "재고 10개 미만 상품 가격 10% 인상" — 변경 감지로 처리하면 **100건이면 UPDATE SQL 100번**.
- 벌크 연산은 **쿼리 한 번으로 여러 테이블 로우 변경**.

```java
// UPDATE/DELETE 는 결과 타입 지정 X (createQuery(jpql, X.class) 금지), getResultList() X
int resultCount = em.createQuery(
        "update Product p set p.price = cast(p.price * 1.1 as Integer) " +
        "where p.stockAmount < :stockAmount")
        .setParameter("stockAmount", 10)
        .executeUpdate();   // WHERE 조건에 걸려 갱신된 row 수 반환
```

- `UPDATE`, `DELETE` 지원 (+`INSERT INTO ... SELECT`는 하이버네이트 지원)
- 직접 틀려본 것들:
  - `createQuery(jpql, Product.class)` → `IllegalQueryOperationException: Result type given for a non-SELECT Query`.
    벌크 연산은 **타입 없이 만들고 `executeUpdate()`** 로 실행한다.
  - ⚠️ Hibernate 6 차이 — `set p.price = p.price * 1.1` 은 Double 을 Integer 필드에 대입한다며
    `SemanticException` 으로 거부된다(강의 버전에선 통과). **`cast(... as Integer)`** 필요.
    참고로 cast 는 DB 에서 실행되므로 소수점 처리도 DB 규칙을 따른다 (H2 는 반올림: 12345×1.1 → 13580).
  - `resultCount` 는 "SET 으로 바뀐 칼럼 수"가 아니라 **WHERE 에 걸린 row 수**다.
    WHERE 에만 쓰인 칼럼(stockAmount)은 당연히 값이 그대로다.

### ⚠️ 벌크 연산 주의
- 벌크 연산은 **영속성 컨텍스트를 무시하고 DB에 직접 쿼리**한다.
  → 영속성 컨텍스트의 엔티티와 DB 상태가 어긋날 수 있다.
- 해결책 두 가지:
  1. **벌크 연산을 가장 먼저 실행**
  2. **벌크 연산 수행 후 영속성 컨텍스트 초기화** (`em.clear()`)

## ⚠️ 강의(Hibernate 5)와 내 환경(Hibernate 6.4)의 차이 모아보기

| 항목 | 강의 (Hibernate 5) | Hibernate 6 (직접 확인) |
|------|--------------------|------------------------|
| 컬렉션 페치 조인 중복 | `distinct` 없으면 팀A 가 2번 나옴 | **자동 중복 제거** — distinct 없이도 1번 |
| `@BatchSize` IN 절 | 100 → 50 → 25 ... 하위 배치로 분할 | **size 만큼 물음표 패딩** (남는 칸은 마지막 id 반복) — SQL 모양 고정으로 실행 계획 재사용 |
| 벌크 UPDATE 타입 검사 | `p.price * 1.1` (Double→Integer) 통과 | `SemanticException` 거부 → **`cast(... as Integer)` 필요** |
| FROM 절 서브쿼리 | 미지원 (조인/네이티브로 우회) | 지원 |
| 패키지 | `javax.persistence.*` | `jakarta.persistence.*` |

## ✅ 핵심 요약 한 줄 정리
1. **묵시적 조인 금지, 명시적 조인 사용** — 컬렉션 경로는 탐색의 끝.
2. **페치 조인** = N+1 해결의 핵심. 단, 별칭 금지(객체 그래프는 "전부"가 전제), 컬렉션 페치 조인 + 페이징 금지, 둘 이상 컬렉션 금지.
3. 컬렉션 + 페이징은 **지연 로딩 + `@BatchSize`(컬렉션 필드에!)** 로 — N+1 이 IN 쿼리 1방이 된다.
4. **JPQL 은 항상 SQL 을 발사** — 1차 캐시 조회 생략은 `em.find()` 전용, JPQL 은 결과 조립 때만 캐시 우선.
5. **Named 쿼리**는 로딩 시점에 문법 검증되는 정적 쿼리. XML 이 어노테이션보다 우선, `<mapping-file>` 은 `<properties>` 앞에.
6. **벌크 연산**은 타입 없이 `createQuery` + `executeUpdate()`, 영속성 컨텍스트를 무시 → 먼저 실행하거나 실행 후 `em.clear()`.
