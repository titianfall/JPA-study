# 04. 회원 도메인 개발

> 현재 `study/SpringBootJPA/jpashop`의 회원 기능 구현을 기준으로 정리한다.
> Spring Boot 3.x, JPA `jakarta.persistence`, JUnit 5 기준

---

## 1. 회원 기능의 역할 분리

회원 가입과 조회 기능은 세 계층으로 나눈다.

| 계층 | 현재 클래스 | 책임 |
| --- | --- | --- |
| 도메인 | `domain.Member` | 회원의 식별자, 이름, 주소, 주문 관계를 표현한다. |
| 리포지토리 | `repository.MemberRepository` | `EntityManager`로 회원을 저장하고 조회한다. |
| 서비스 | `service.MemberService` | 중복 검사, 트랜잭션, 회원가입 유스케이스를 담당한다. |

컨트롤러는 이후 웹 계층에서 요청과 응답을 처리하고, 회원가입 규칙은 서비스에 둔다. 이렇게 하면 화면이나 API가 바뀌어도 핵심 도메인 로직을 재사용할 수 있다.

---

## 2. `MemberRepository`: 영속성 접근만 담당

```java
@Repository
@RequiredArgsConstructor
public class MemberRepository {

    private final EntityManager em;

    public void save(Member member) {
        em.persist(member);
    }

    public Member findOne(Long id) {
        return em.find(Member.class, id);
    }

    public List<Member> findByName(String name) {
        return em.createQuery(
                        "select m from Member m where m.name = :name", Member.class)
                .setParameter("name", name)
                .getResultList();
    }
}
```

- `@Repository`는 스프링 빈으로 등록하고 영속성 예외 변환을 적용한다.
- `@RequiredArgsConstructor`가 `final EntityManager` 생성자를 만들어 주므로, 생성자가 하나인 현재 클래스에서는 `@Autowired`를 생략해도 생성자 주입된다.
- `persist()`는 새 엔티티를 영속 상태로 만든다. 실제 INSERT 시점은 ID 생성 전략과 flush 시점에 따라 달라질 수 있다.
- 이름은 유일 제약 조건으로 DB에도 두는 것이 안전하지만, 서비스에서 먼저 검사해 사용자에게 의미 있는 예외를 제공한다.

> `EntityManager`는 `@PersistenceContext` 필드 주입으로도 받을 수 있다. 현재 구현은 `final` 필드를 사용할 수 있는 생성자 주입을 선택했다.

## 3. `MemberService`: 유스케이스와 트랜잭션

```java
@Service
@Transactional(readOnly = true)
@RequiredArgsConstructor
public class MemberService {

    private final MemberRepository memberRepository;

    @Transactional
    public Long join(Member member) {
        validateDuplicateMember(member);
        memberRepository.save(member);
        return member.getId();
    }

    private void validateDuplicateMember(Member member) {
        List<Member> members = memberRepository.findByName(member.getName());
        if (!members.isEmpty()) {
            throw new IllegalStateException("이미 존재하는 회원입니다.");
        }
    }
}
```

클래스의 `@Transactional(readOnly = true)`는 조회 메서드의 기본값이다. 회원가입처럼 데이터를 바꾸는 메서드에만 `@Transactional`을 다시 선언해 쓰기 트랜잭션을 적용한다.

`join()`은 저장한 `Member` 객체 대신 ID를 반환한다. 호출부가 필요한 경우 ID로 다시 조회하게 하여 저장 명령과 조회를 구분한다(CQS).

### 동시성 주의

애플리케이션 수준의 중복 검사만으로는 동시에 들어온 두 요청을 완전히 막을 수 없다. 실제 서비스에서는 `member.name`에 유니크 제약 조건을 추가하고, 제약 조건 위반도 적절한 예외로 변환해야 한다.

## 4. 회원 기능 테스트

```java
@SpringBootTest
@Transactional
class MemberServiceTest {

    @Autowired MemberService memberService;
    @Autowired MemberRepository memberRepository;

    @Test
    void 회원가입() {
        Member member = new Member();
        member.setName("kim");

        Long savedId = memberService.join(member);

        assertEquals(member, memberRepository.findOne(savedId));
    }

    @Test
    void 중복_회원_예외() {
        Member member1 = new Member();
        member1.setName("kim");
        Member member2 = new Member();
        member2.setName("kim");

        memberService.join(member1);

        assertThrows(IllegalStateException.class,
                () -> memberService.join(member2));
    }
}
```

- `@SpringBootTest`는 스프링 컨테이너를 띄워 서비스·리포지토리·JPA를 함께 검증한다.
- 테스트 클래스의 `@Transactional`은 테스트가 끝날 때 기본적으로 롤백한다. INSERT 로그가 보여도 마지막에 rollback되면 데이터는 남지 않는다.
- JUnit 5에서는 예외 검증에 `assertThrows()`를 사용한다. `try-catch + return + fail()` 또는 JUnit 4의 `@Test(expected = ...)`는 현재 프로젝트에 필요 없다.
- `assertThrows()`가 기대한 예외를 검증한 뒤에는 다음 줄을 계속 실행한다. 따라서 그 아래에 `fail()`을 두면 테스트가 실패하므로 함께 사용하지 않는다.

## 5. 테스트 DB

`src/test/resources/application.yml`에는 H2 인메모리 DB 예시가 준비되어 있다.

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:test
  jpa:
    hibernate:
      ddl-auto: create-drop
```

현재 파일에서는 데이터소스 설정이 주석 처리되어 있으므로, 테스트는 기본 `application.yml`의 H2 TCP 설정을 사용한다. 테스트를 외부 H2 서버와 분리하려면 위 설정의 주석을 해제한다.

## ✅ 핵심 요약

1. 리포지토리는 `EntityManager`를 사용한 저장·조회만 담당한다.
2. 서비스는 중복 검사와 트랜잭션을 포함한 회원가입 유스케이스를 담당한다.
3. 기본 조회 트랜잭션은 `readOnly = true`, 데이터 변경 메서드는 별도 `@Transactional`로 선언한다.
4. JUnit 5 예외 테스트는 `assertThrows()`로 작성하고, 뒤에 `fail()`을 두지 않는다.
5. 테스트의 롤백은 INSERT SQL의 실행 여부와 별개로 최종 DB 변경을 되돌린다.
