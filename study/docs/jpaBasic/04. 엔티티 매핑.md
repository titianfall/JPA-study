## 1. 객체와 테이블 매핑 — @Entity / @Table

### @Entity

- `@Entity`가 붙은 클래스는 **JPA가 관리**한다. JPA로 테이블과 매핑할 클래스는 필수.
- 내 `Member.java`:

```java
// jpa 가 관리하는 entity로 인식 및 관리한다.
// 일반적으로 클래스 이름 = 테이블 이름 매핑
@Entity
public class Member {

    @Id
    private long id;
    private String name;

    // 기본 생성자가 필요하다. - 추후 학습
    public Member() {
    }
    ...
}
```

⚠️ **@Entity 적용 시 주의사항** (내 코드 주석의 "기본 생성자가 필요하다"가 여기 해당):

1. **기본 생성자 필수** — 파라미터 없는 `public` 또는 `protected` 생성자
   (JPA가 리플렉션으로 객체를 생성/프록시를 만들 때 필요)
2. `final` 클래스, `enum`, `interface`, `inner` 클래스에는 사용 불가
3. 저장할 필드에 `final` 사용 불가

- 속성 `name`: JPA에서 사용할 엔티티 이름. 기본값은 클래스 이름(`Member`). 특별한 이유 없으면 기본값 사용.

### @Table

- 엔티티와 매핑할 **테이블**을 지정 (클래스명과 테이블명이 다를 때)

| 속성 | 기능 |
|------|------|
| `name` | 매핑할 테이블 이름 (기본값: 엔티티 이름) |
| `catalog` / `schema` | DB catalog / schema 매핑 |
| `uniqueConstraints` | DDL 생성 시 유니크 제약조건 생성 |

```java
@Entity
@Table(name = "MBR")   // 이러면 쿼리가 MBR 테이블로 나간다
public class Member { ... }
```

## 2. 데이터베이스 스키마 자동 생성 — hibernate.hbm2ddl.auto

- 애플리케이션 실행 시점에 **DDL을 자동 생성**해준다. 테이블 중심 → **객체 중심** 개발 가능.
- DB **방언(Dialect)에 맞는 적절한 DDL** 생성 (H2/MySQL/Oracle 각각 다르게).

| 옵션 | 설명 |
|------|------|
| `create` | 기존 테이블 삭제 후 다시 생성 (DROP + CREATE) |
| `create-drop` | create + 애플리케이션 종료 시점에 테이블 DROP |
| `update` | 변경분만 반영 (컬럼 추가만 됨, **삭제는 안 됨**) |
| `validate` | 엔티티와 테이블이 정상 매핑되었는지만 확인 |
| `none` | 사용하지 않음 |

내 `persistence.xml`에는 현재 주석 처리되어 있다 (테이블을 직접 만들어서 실습 중):

```xml
<!--<property name="hibernate.hbm2ddl.auto" value="create" />-->
```

⚠️ **운영 장비에는 절대 create, create-drop, update 사용하면 안 된다!**

| 환경 | 권장 옵션 |
|------|-----------|
| 개발 초기 (로컬) | `create` 또는 `update` |
| 테스트 서버 (여럿이 공유) | `update` 또는 `validate` |
| 스테이징 / 운영 | `validate` 또는 `none` |

> 💡 운영 DB에서 `create`면 데이터 전체가 날아가고, `update`도 ALTER TABLE이
> 테이블 락을 걸어 서비스가 멈출 수 있다. 결국 운영에서는 직접 SQL을 다듬어 적용하는 게 안전.

### DDL 생성 기능

```java
@Column(nullable = false, length = 10)   // NOT NULL, VARCHAR(10)
private String name;

@Table(uniqueConstraints = {@UniqueConstraint(
        name = "NAME_AGE_UNIQUE",
        columnNames = {"NAME", "AGE"} )})
```

- 이런 제약조건은 **DDL을 자동 생성할 때만 사용**되고, **JPA의 실행 로직(SQL 생성)에는 영향을 주지 않는다.**

## 3. 필드와 컬럼 매핑

강의 예제 (요구사항: 회원은 일반/관리자 구분, 가입일·수정일, 자유 설명 필드):

```java
@Entity
public class Member {

    @Id
    private Long id;

    @Column(name = "name")            // 객체는 username, DB 컬럼은 name
    private String username;

    private Integer age;              // 가장 적절한 숫자 타입으로 알아서 매핑

    @Enumerated(EnumType.STRING)      // enum 매핑
    private RoleType roleType;

    @Temporal(TemporalType.TIMESTAMP) // 날짜 매핑 (구버전 방식)
    private Date createdDate;

    @Temporal(TemporalType.TIMESTAMP)
    private Date lastModifiedDate;

    @Lob                              // 큰 텍스트 → CLOB
    private String description;

    @Transient                        // DB 매핑 제외, 메모리에서만
    private Integer temp;
}
```

> ⚠️ 강의 PDF는 `import javax.persistence.*;` 지만 내 프로젝트(Hibernate 6.x)는
> `jakarta.persistence.*` 를 사용한다.

### 매핑 어노테이션 정리

| 어노테이션 | 설명 |
|-----------|------|
| `@Column` | 컬럼 매핑 |
| `@Temporal` | 날짜 타입 매핑 (`Date`, `Calendar` 용) |
| `@Enumerated` | enum 타입 매핑 |
| `@Lob` | BLOB, CLOB 매핑 |
| `@Transient` | 특정 필드를 컬럼에 매핑하지 않음 |

### @Column 주요 속성

| 속성 | 설명 | 기본값 |
|------|------|--------|
| `name` | 매핑할 컬럼 이름 | 필드 이름 |
| `insertable`, `updatable` | 등록/변경 가능 여부 | `true` |
| `nullable` (DDL) | `false`면 NOT NULL 제약조건 | `true` |
| `unique` (DDL) | 유니크 제약 (이름이 랜덤 생성돼서 잘 안 씀 → `@Table`의 `uniqueConstraints` 사용) | |
| `columnDefinition` (DDL) | 컬럼 정보 직접 지정. 예: `varchar(100) default 'EMPTY'` | |
| `length` (DDL) | 문자 길이 제약 (String만) | 255 |
| `precision`, `scale` (DDL) | `BigDecimal`용. 전체 자릿수 / 소수 자릿수 | 19, 2 |

### @Enumerated — ⚠️ ORDINAL 절대 금지

- `EnumType.ORDINAL` (기본값): enum **순서**를 숫자로 저장
- `EnumType.STRING`: enum **이름**을 문자열로 저장

⚠️ **반드시 `STRING` 사용!** ORDINAL로 저장하면, 나중에 enum 앞쪽에 상수를 추가했을 때
기존 데이터의 숫자 의미가 전부 바뀌어버린다 (0번이 USER였는데 GUEST가 됨).
이미 저장된 데이터는 안 바뀌므로 **데이터가 통째로 꼬이는 대형 사고**.

### @Temporal — 최신 버전에서는 사실상 불필요

- `java.util.Date`, `java.util.Calendar` 매핑용 (DATE / TIME / TIMESTAMP)
- 💡 **`LocalDate`, `LocalDateTime`을 쓰면 생략 가능** (최신 하이버네이트가 자동 매핑)

```java
private LocalDate testLocalDate;         // → date 타입
private LocalDateTime testLocalDateTime; // → timestamp 타입
```

### @Lob

- 지정할 속성이 없다. 필드 타입이 문자면 **CLOB**, 나머지는 **BLOB**으로 매핑
  - CLOB: `String`, `char[]` / BLOB: `byte[]`

### @Transient

- DB 저장 X, 조회 X. **메모리에서만 임시로 쓸 값**일 때 사용

## 4. 기본 키 매핑 — @Id / @GeneratedValue

```java
@Id @GeneratedValue(strategy = GenerationType.AUTO)
private Long id;
```

- **직접 할당**: `@Id`만 사용 — 지금 내 코드가 이 방식 (`new Member(150L, "A")` 처럼 id를 직접 넣는다)
- **자동 생성**: `@GeneratedValue` + 전략 선택

| 전략 | 방식 | 주 사용 DB |
|------|------|-----------|
| `IDENTITY` | 기본 키 생성을 DB에 위임 (AUTO_INCREMENT) | MySQL, PostgreSQL |
| `SEQUENCE` | DB 시퀀스 오브젝트 사용 | Oracle, H2, PostgreSQL |
| `TABLE` | 키 생성 전용 테이블로 시퀀스 흉내 | 모든 DB |
| `AUTO` | 방언에 따라 위 셋 중 자동 선택 (기본값) | - |

### IDENTITY 전략 — 쓰기 지연이 안 통하는 예외

```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

- id 값을 **DB에 INSERT 해봐야** 알 수 있다 (AUTO_INCREMENT는 DB가 채번).
- 그런데 3장에서 배웠듯 영속성 컨텍스트는 **id를 키로 1차 캐시에 엔티티를 관리**한다.
  → id를 모르면 영속 상태로 만들 수가 없다!
- 그래서 IDENTITY 전략만 예외적으로 **`em.persist()` 시점에 즉시 INSERT SQL 실행**
  (트랜잭션 커밋까지 미루는 **쓰기 지연이 동작하지 않는다**)

### SEQUENCE 전략

```java
@Entity
@SequenceGenerator(
        name = "MEMBER_SEQ_GENERATOR",
        sequenceName = "MEMBER_SEQ",      // 매핑할 DB 시퀀스 이름
        initialValue = 1, allocationSize = 1)
public class Member {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE,
            generator = "MEMBER_SEQ_GENERATOR")
    private Long id;
```

- `em.persist()` 시점에 DB 시퀀스에서 **다음 값만 조회**해서 1차 캐시에 넣는다
  → INSERT는 커밋까지 미룰 수 있어 **쓰기 지연이 그대로 동작**

💡 **allocationSize = 50 (기본값)의 의미 — 성능 최적화**
- persist 할 때마다 시퀀스를 조회하면 네트워크 왕복이 매번 발생
- `allocationSize = 50`이면 시퀀스를 **한 번 호출해 50개 범위를 미리 확보**하고,
  메모리에서 1씩 꺼내 쓴다 (50개 소진 시 다시 호출)
- 여러 서버가 동시에 써도 각자 다른 범위를 받아가므로 충돌 없음

### TABLE 전략

- 키 생성 전용 테이블(`sequence_name`, `next_val` 컬럼)을 만들어 시퀀스를 흉내
- 장점: 모든 DB에서 동작 / 단점: 테이블 락 등 **성능 문제** → 실무에서는 잘 안 씀

### 권장 식별자 전략

- 좋은 기본 키 조건: **null 아님, 유일, 변하면 안 됨**
- 주민번호·전화번호 같은 **자연 키(natural key)는 기본 키로 쓰지 말 것**
  → 비즈니스 규칙은 언젠가 바뀐다 (예: 주민번호 저장 금지 정책)
- ✅ **권장: `Long` 타입 + 대체 키(auto increment / sequence) + 키 생성전략 사용**

> ⚠️ 내 `Member.java`는 현재 `private long id` (프리미티브)인데,
> `@GeneratedValue`를 쓰려면 **`Long` (래퍼 타입)** 으로 바꿔야 한다.
> 프리미티브 `long`은 null이 없어서 "아직 id가 없는 상태(비영속)"를 표현할 수 없기 때문.
> (02장에서 정리한 `long id` vs `Long id` 차이가 여기서 실제로 문제가 된다)

## 5. 실전 예제 1 — 요구사항 분석과 기본 매핑

**요구사항**: 회원은 상품을 주문할 수 있다 / 주문 시 여러 종류의 상품을 선택할 수 있다

- 도메인: 회원(Member) — 주문(Order) — 주문상품(OrderItem) — 상품(Item)
- 회원 기능: 등록/조회, 상품 기능: 등록/수정/조회, 주문 기능: 주문/내역조회/취소

### 요구사항 → 모델링 → 테이블 설계 흐름

1. **요구사항 분석**
   - 회원은 여러 번 주문할 수 있다 → `Member 1 : N Order`
   - 주문할 때 여러 상품을 선택할 수 있다 → `Order N : M Item`
   - 같은 상품도 여러 주문에 포함될 수 있다 → 상품 입장에서도 주문과 다대다

2. **개념적 모델링**
   - 처음에는 `회원 - 주문 - 상품`으로 보이지만, `주문`과 `상품`은 다대다 관계다.
   - 다대다는 테이블로 바로 풀기 어렵고, 주문 수량/주문 가격 같은 추가 정보도 필요하다.
   - 그래서 중간 모델 `주문상품(OrderItem)`을 둔다.

```text
Member 1 ── N Order
Order  1 ── N OrderItem
Item   1 ── N OrderItem
```

3. **논리적 모델링**

| 엔티티 | 주요 속성 | 관계 |
|--------|-----------|------|
| `Member` | `id`, `name`, `city`, `street`, `zipcode` | 주문을 여러 개 가짐 |
| `Order` | `id`, `memberId`, `orderDate`, `status` | 회원 1명에게 속함 |
| `Item` | `id`, `name`, `price`, `stockQuantity` | 여러 주문상품에 포함됨 |
| `OrderItem` | `id`, `orderId`, `itemId`, `orderPrice`, `count` | 주문과 상품을 연결 |

4. **최종 테이블 설계**
   - 1:N 관계는 **N쪽 테이블이 FK를 가진다**. 예: `ORDERS.MEMBER_ID`
   - N:M 관계는 **중간 테이블로 풀어서 1:N + N:1로 만든다**. 예: `ORDER_ITEM`

```text
MEMBER
- MEMBER_ID PK

ORDERS
- ORDER_ID PK
- MEMBER_ID FK

ITEM
- ITEM_ID PK

ORDER_ITEM
- ORDER_ITEM_ID PK
- ORDER_ID FK
- ITEM_ID FK
- ORDER_PRICE
- COUNT
```

> 이 장에서는 아직 연관관계 매핑 전이라 `memberId`, `orderId`, `itemId`처럼
> 외래 키 값을 필드로 직접 들고 간다. 다음 장에서 `Member member`, `Order order`,
> `Item item` 같은 객체 참조로 바꾼다.

### 데이터 중심 설계의 문제점 (다음 장 예고)

이 장까지의 매핑은 외래 키를 필드로 그대로 들고 있는 **데이터(테이블) 중심 설계**:

```java
@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;

    @Column(name = "MEMBER_ID")
    private Long memberId;   // ← 객체가 아니라 외래 키 값을 그대로 보관
    ...
}
```

- 주문한 회원을 찾으려면 `order.getMemberId()` → `em.find(Member.class, memberId)` 를 **직접 두 번**
  해야 한다. 객체 그래프 탐색(`order.getMember()`)이 불가능 → **객체지향스럽지 않다**
- 이 문제를 푸는 것이 다음 장 **연관관계 매핑**(`@ManyToOne`, `@JoinColumn`)

## ✅ 핵심 요약 한 줄 정리

1. **@Entity** = JPA 관리 대상. **기본 생성자 필수**, final 클래스/필드 X
2. **hbm2ddl.auto**로 DDL 자동 생성 — 로컬은 `create`/`update`, **운영은 절대 금지** (`validate`/`none`)
3. DDL 제약조건(`nullable`, `length` 등)은 **DDL 생성에만 영향**, JPA 실행 로직과 무관
4. **@Enumerated는 무조건 `STRING`** (ORDINAL은 enum 순서 변경 시 데이터 대참사)
5. 날짜는 `LocalDate`/`LocalDateTime` 쓰면 `@Temporal` 생략 가능
6. **IDENTITY**는 persist 즉시 INSERT(쓰기 지연 X), **SEQUENCE**는 채번만 하고 쓰기 지연 O (+ `allocationSize` 50 최적화)
7. 기본 키는 **`Long` + 대체 키 + 생성 전략** (자연 키 금지)
8. 외래 키를 필드로 들고 있는 설계는 객체 그래프 탐색이 안 됨 → 5장 연관관계 매핑으로
