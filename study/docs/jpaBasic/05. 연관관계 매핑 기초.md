## 0. 용어부터 정리

| 용어 | 의미 |
|------|------|
| **방향** (Direction) | 단방향, 양방향 |
| **다중성** (Multiplicity) | 다대일(N:1), 일대다(1:N), 일대일(1:1), 다대다(N:M) |
| **연관관계의 주인** (Owner) | 객체 양방향 연관관계에서 **외래 키를 관리**하는 쪽 |

## 1. 연관관계가 필요한 이유

### 내 코드가 딱 이 상태였다 (객체를 테이블에 맞춘 모델링)

4장 실전 예제에서 작성한 내 `Order.java` — 참조 대신 **외래 키 값을 그대로 필드로** 들고 있다:

```java
@Entity
@Table(name = "ORDERS")
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.AUTO)
    @Column(name = "ORDER_ID")
    private Long id;

    @Column(name = "MEMBER_ID")
    private Long memberId;   // 참조가 아니라 FK 식별자를 직접 보관

    // private Member member;   // ← 5장에서 이걸 살려서 연관관계로 바꾼다
    ...
}
```

그래서 조회할 때 내 `JpaMain.java`처럼 **식별자로 다시 조회**해야 한다:

```java
// 1차로 주문 id를 찾고, 그걸 통해 또 탐색해야 한다.
Order order = em.find(Order.class, 1L);
Long memberId = order.getMemberId();

em.find(Member.class, memberId);
// 객체지향 프로그램을 만들고 있는데 전혀 객체지향적이지 않다.

// Member findMember = order.getMember();  // ← 이렇게 탐색하고 싶다
```

### 왜 이런 간격이 생기나

- **테이블**은 **외래 키로 조인**해서 연관된 테이블을 찾는다.
- **객체**는 **참조**를 사용해서 연관된 객체를 찾는다 (`order.getMember()`).
- 객체를 테이블에 맞춰 데이터 중심으로 모델링하면 **객체 간 협력 관계를 만들 수 없다.**

### Member–Team으로 직접 재현해보기 (5장 실습 출발점)

강의 시나리오(회원 N : 팀 1)를 일부러 **객체를 테이블에 맞춘 방식**으로 먼저 작성했다.
내 `Member.java` — 참조 대신 FK 식별자를 그대로 필드로:

```java
@Entity
public class Member {
    @Id @GeneratedValue(strategy = GenerationType.AUTO)
    @Column(name = "MEMBER_ID")
    private Long id;

    @Column(name = "USERNAME")
    private String username;

    @Column(name = "TEAM_ID")
    private Long teamId;    // ❌ Team이 아니라 그냥 숫자(FK 값)
    ...
}
```

내 `JpaMain.java` — 저장도 조회도 식별자를 손으로 나른다:

```java
// 외래 키 식별자를 직접 다룸
member.setTeamId(team.getId()); // member.setTeam만 하면 될 것 같은데...
em.persist(member);

// 조회 - 연관관계가 없기 때문에 식별자로 다시 조회
Member findMember = em.find(Member.class, member.getId());
Team findTeam = em.find(Team.class, team.getId());
```

💡 이게 귀찮은 이유: `em.find`가 그래프 탐색을 "못 해줘서"가 아니라,
`Long teamId`에는 따라갈 **참조(간선)가 없어서 객체 그래프가 아예 안 만들어져 있기 때문**.
DB가 FK로 조인하는 일을 개발자가 손으로 번역하고 있는 셈이다.
→ 다음 절에서 `teamId`를 지우고 `Team team` 참조로 바꾼다.

## 2. 단방향 연관관계

강의 예제 시나리오: 회원(Member)과 팀(Team), 회원은 하나의 팀에만 소속 → **다대일(N:1)**.

FK 필드를 지우고 **참조 + 매핑 어노테이션**으로 바꾼다:

```java
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;

    // @Column(name = "TEAM_ID")
    // private Long teamId;         // ❌ 외래 키 식별자 직접 보관

    @ManyToOne                      // 다중성: Member(N) : Team(1)
    @JoinColumn(name = "TEAM_ID")   // 조인할 FK 컬럼명
    private Team team;              // ✅ 참조로 연관관계
    ...
}
```

- **`@ManyToOne`** : 다중성. "내가(Member) N이고 상대(Team)가 1" — 필드 기준으로 읽으면 됨.
- **`@JoinColumn(name = "TEAM_ID")`** : 이 참조가 매핑될 **외래 키 컬럼** 지정.

### 저장 / 조회 / 수정 — 전부 참조로

```java
// 저장: FK 값이 아니라 팀 객체 자체를 넣는다
Team team = new Team();
team.setName("TeamA");
em.persist(team);

Member member = new Member();
member.setName("member1");
member.setTeam(team);        // 단방향 연관관계 설정, 참조 저장
em.persist(member);

// 조회: 객체 그래프 탐색
Member findMember = em.find(Member.class, member.getId());
Team findTeam = findMember.getTeam();   // ✅ 참조로 바로 탐색

// 수정: 다른 팀 객체로 갈아끼우면 UPDATE 시 FK가 바뀐다
member.setTeam(teamB);
```

💡 저장 시 JPA가 `team` 참조에서 PK를 꺼내 `TEAM_ID` FK로 INSERT 해준다. 조회 시엔 조인해서 객체로 만들어준다. **패러다임 번역을 JPA가 대신하는 것.**

### 직접 실행하며 관찰한 것들 (Hibernate 로그)

```
create table Member (... TEAM_ID bigint ...)   ① 테이블 전부 생성 (FK 없이)
create table Team (...)
alter table Member add constraint ...          ② FK는 마지막에 alter로 일괄 추가
   foreign key (TEAM_ID) references Team

select next value for Member_SEQ               ③ persist: INSERT 없이 id 채번만
findTeam = TeamB                               ④ em.find: SELECT 없음 → 1차 캐시 히트
insert into Team ... / insert into Member ...  ⑤ tx.commit 시점에 flush → INSERT
```

- **FK를 `alter table`로 마지막에 거는 이유**: FK는 양쪽 테이블이 모두 있어야 생성 가능. 테이블 먼저 다 만들고 FK를 나중에 걸면 생성 순서 고민이 없고, 순환 참조 구조도 만들 수 있다.
- **`em.persist()` 시점엔 시퀀스 호출만** — 영속 상태에 id가 필요하니 채번만 하고, INSERT는 쓰기 지연 저장소에 모아둔다.
- **방금 persist한 엔티티의 `em.find`는 flush 없이 1차 캐시에서 반환** — SELECT도 INSERT도 안 나간다. (JPQL이었다면 쿼리 직전 자동 flush)

⚠️ `em.find(Member.class, member.getTeam())` 처럼 두 번째 인자에 연관 객체를 넣으면 `Cannot coerce Team to Long` 예외. `em.find(엔티티, PK)`로 **주인 엔티티를 꺼낸 뒤** `getTeam()`으로 그래프를 탐색하는 것이 정답 형태.

### `em.find` 시 left (equi) join이 나가는 이유

```sql
select m.*, t.*
from Member m
left join Team t on t.TEAM_ID = m.TEAM_ID   -- left outer + equi join
where m.MEMBER_ID = ?
```

- **join을 하는 이유**: `@ManyToOne`의 기본 fetch가 **EAGER** — "Member 조회 시 Team도 즉시 채워라". PK 단건 조회에선 쿼리 2번 대신 **조인 한 방**으로 최적화한다.
- **하필 left join인 이유**: `TEAM_ID`가 nullable이라 **팀 없는 회원**이 있을 수 있다. inner join이면 그런 회원은 행 자체가 사라져 `em.find`가 null을 반환하는 참사. left join이면 회원은 살아있고 `getTeam()`만 null.
- `@ManyToOne(optional = false)` 등으로 not null을 보장하면 inner join으로 바뀔 수 있다.

⚠️ 실무에서는 EAGER 기본값이 골칫거리(의도 안 한 조인, N+1) → **`@ManyToOne(fetch = FetchType.LAZY)`가 실무 표준**. 자세한 건 8장(프록시와 연관관계 관리)에서.

## 3. 양방향 연관관계 — Team에서도 회원들을 보고 싶다

`Member → Team` 단방향에, 반대 방향 `Team → Member`를 추가:

```java
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team")   // Member.team이 주인임을 명시
    List<Member> members = new ArrayList<Member>();
    ...
}
```

```java
// 반대 방향으로 객체 그래프 탐색
Team findTeam = em.find(Team.class, team.getId());
int memberSize = findTeam.getMembers().size();   // 역방향 조회
```

⚠️ **테이블은 바뀌는 게 없다.** `MEMBER.TEAM_ID` FK 하나로 이미 양쪽 조인이 다 된다. 양방향이 필요한 건 **객체** 쪽 사정이다.

---

## 4. mappedBy와 연관관계의 주인 (⭐ 이 장의 핵심)

### 객체와 테이블이 관계를 맺는 방식의 차이

| | 연관관계 개수 | 방식 |
|---|---|---|
| **객체** | **2개** (Member→Team, Team→Member) | 서로 다른 **단방향 참조 2개** |
| **테이블** | **1개** (MEMBER ↔ TEAM) | **외래 키 하나**로 양쪽 조인 |

객체의 "양방향"은 사실 양방향이 아니라 **단방향 관계 2개**다. 그런데 테이블의 FK는 하나 →
`member.setTeam()`과 `team.getMembers().add()` 중 **누가 FK를 바꿀지** 정해야 한다.

### 양방향 매핑 규칙

- 두 관계 중 하나를 **연관관계의 주인(Owner)** 으로 지정
- **주인만이 외래 키를 관리** (등록, 수정)
- 주인이 아닌 쪽은 **읽기만 가능**
- 주인은 `mappedBy` 속성을 **쓰지 않고**, 주인이 아니면 `mappedBy`로 주인을 지정

### 누구를 주인으로?

> **외래 키가 있는 곳을 주인으로 정해라.** (= N쪽, `@ManyToOne` 쪽)

- 여기서는 FK(`TEAM_ID`)가 MEMBER 테이블에 있으므로 **`Member.team`이 주인**.
- ⚠️ **비즈니스 로직 기준으로 주인을 정하면 안 된다.** 오직 **외래 키의 위치** 기준.
  주인이 아닌 쪽(`Team.members`)을 바꿔도 DB에 아무 일도 안 일어나는 게 직관과 어긋나지 않으려면, "FK 있는 곳 = 주인"이 제일 단순하다.

## 5. 양방향 매핑 시 가장 많이 하는 실수

### 실수: 주인이 아닌 방향에만 값을 입력

```java
Team team = new Team();
team.setName("TeamA");
em.persist(team);

Member member = new Member();
member.setName("member1");
team.getMembers().add(member);   // ❌ 역방향(주인 아님)에만 설정
em.persist(member);
```

| ID | USERNAME | TEAM_ID |
|----|----------|---------|
| 1 | member1 | **null** |

`Team.members`는 읽기 전용이라 **FK에 반영되지 않는다** → `TEAM_ID`가 null.

### 정답: 주인에 값을 설정 (순수 객체를 고려하면 양쪽 다)

```java
team.getMembers().add(member);   // (객체 상태를 위해)
member.setTeam(team);            // ✅ 연관관계의 주인에 값 설정 — 이게 FK를 만든다
em.persist(member);
```

💡 **연관관계 편의 메소드**를 만들어 한 번에 양쪽을 세팅하면 실수를 막을 수 있다:

```java
// Member 쪽에 (한쪽에만 만들 것)
public void changeTeam(Team team) {
    this.team = team;
    team.getMembers().add(this);
}
```

⚠️ **무한 루프 조심**: `toString()`(Lombok), JSON 생성 라이브러리가 양쪽을 서로 호출하면 스택오버플로우. 엔티티는 컨트롤러에서 직접 반환하지 말고 DTO로 변환해서 반환하자.

## 6. 양방향 매핑 정리

- **단방향 매핑만으로 이미 연관관계 매핑은 완료**된 것이다.
- 양방향은 거기에 **반대 방향 조회(객체 그래프 탐색) 기능이 추가**된 것뿐.
- 설계 단계에서는 **단방향으로 끝내고**, 역방향 탐색이 필요해지면(JPQL에서 많음) 그때 추가해도 된다 — **테이블에는 영향이 없기** 때문.

## 7. 실전 예제 2 — jpashop에 연관관계 매핑 적용

테이블 구조는 4장과 동일. **객체가 참조를 사용하도록** 바꾼다:

| 엔티티 | 4장 (FK 식별자) | 5장 (참조) |
|--------|----------------|-----------|
| `Order` | `Long memberId` | `@ManyToOne @JoinColumn(name = "MEMBER_ID") Member member` |
| `OrderItem` | `Long orderId` | `@ManyToOne @JoinColumn(name = "ORDER_ID") Order order` |
| `OrderItem` | `Long itemId` | `@ManyToOne @JoinColumn(name = "ITEM_ID") Item item` |
| `Order` (양방향, 선택) | — | `@OneToMany(mappedBy = "order") List<OrderItem> orderItems` |

- 주인은 전부 **FK를 가진 N쪽** (`Order.member`, `OrderItem.order`, `OrderItem.item`).
- `Member`에 `List<Order> orders`를 넣는 것은 가능하지만, 회원의 주문은 보통 `Order`에서 조건 조회하는 게 맞아서 **굳이 양방향으로 안 해도 된다** (단방향 우선 원칙).
- 이제 4장에서 주석으로 남겨둔 `order.getMember()` 객체 그래프 탐색이 가능해진다:

```java
Order order = em.find(Order.class, 1L);
Member findMember = order.getMember();   // ✅ 식별자 재조회 없이 참조로 탐색
```

## ✅ 핵심 요약 한 줄 정리

1. FK 식별자를 필드로 들고 다니면 **객체 그래프 탐색이 불가능** → 참조(`@ManyToOne` + `@JoinColumn`)로 매핑
2. 객체의 양방향 = **단방향 2개**, 테이블의 양방향 = **FK 1개** — 이 차이 때문에 주인이 필요
3. **연관관계의 주인 = 외래 키가 있는 곳(N쪽)**, 주인만 FK를 등록/수정, 반대쪽은 `mappedBy`(읽기 전용)
4. 값은 **주인에 반드시 입력** (순수 객체까지 고려해 양쪽 입력, 편의 메소드 활용)
5. `toString()`/JSON 무한 루프 주의 — 엔티티 직접 반환 금지
6. **단방향으로 설계를 끝내고**, 양방향은 필요할 때 추가 (테이블 영향 없음)
