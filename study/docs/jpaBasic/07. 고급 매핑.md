## 0. 왜 상속관계 매핑이 필요한가

- **관계형 데이터베이스에는 상속이라는 개념이 없다.**
- 대신 **슈퍼타입-서브타입 관계**라는 모델링 기법이 객체의 상속과 가장 유사하다.
- 상속관계 매핑 = 객체의 상속 구조와 DB의 슈퍼타입-서브타입 관계를 매핑하는 것.

```
객체:  Item ◁── Album, Movie, Book        (상속)
DB:    ITEM(물품) ─ 슈퍼타입
       ALBUM/MOVIE/BOOK ─ 서브타입         (논리 모델)
```

논리 모델(슈퍼타입-서브타입)을 실제 물리 테이블로 구현하는 방법이 3가지 있고, JPA는 셋 다 지원한다.

| 물리 모델 구현 | JPA 전략 |
|----------------|----------|
| 각각 테이블로 변환 | 조인 전략 (`JOINED`) |
| 통합 테이블로 변환 | 단일 테이블 전략 (`SINGLE_TABLE`) |
| 서브타입 테이블로 변환 | 구현 클래스마다 테이블 전략 (`TABLE_PER_CLASS`) |

중요한 점: **어떤 전략을 골라도 객체 코드(상속 구조)는 거의 그대로다.** 부모 클래스의 애노테이션 한 줄로 전략을 바꿀 수 있다. 이게 JPA의 큰 장점이다.

---

## 1. 주요 어노테이션

```java
@Entity
@Inheritance(strategy = InheritanceType.JOINED)  // 전략 선택 (기본값: SINGLE_TABLE)
@DiscriminatorColumn(name = "DTYPE")             // 부모 테이블에 자식 구분 컬럼 (기본명 DTYPE)
public abstract class Item { ... }

@Entity
@DiscriminatorValue("A")                         // DTYPE에 들어갈 값 (기본값: 엔티티명)
public class Album extends Item { ... }
```

| 어노테이션 | 위치 | 역할 |
|------------|------|------|
| `@Inheritance(strategy = ...)` | 부모 | `JOINED` / `SINGLE_TABLE` / `TABLE_PER_CLASS` 전략 선택 |
| `@DiscriminatorColumn(name = "DTYPE")` | 부모 | 어떤 자식인지 구분하는 컬럼 생성 |
| `@DiscriminatorValue("XXX")` | 자식 | 구분 컬럼에 저장될 값 지정 (기본은 엔티티 이름) |

💡 `@DiscriminatorColumn`은 JOINED 전략에서는 생략해도 동작하지만(조인으로 구분 가능), **DB만 보고 어떤 타입인지 알 수 있도록 넣어주는 것이 운영상 좋다.** SINGLE_TABLE 전략에서는 생략해도 필수라서 자동 생성된다.

---

## 2. 조인 전략 (JOINED) - 기본으로 선택할 전략

부모, 자식을 **각각 테이블로** 만들고, 자식 테이블이 부모의 PK를 자신의 PK 겸 FK로 사용한다.

```
ITEM (ITEM_ID PK, NAME, PRICE, DTYPE)
  ├─ ALBUM (ITEM_ID PK/FK, ARTIST)
  ├─ MOVIE (ITEM_ID PK/FK, DIRECTOR, ACTOR)
  └─ BOOK  (ITEM_ID PK/FK, AUTHOR, ISBN)
```

```java
@Entity
@Inheritance(strategy = InheritanceType.JOINED)
@DiscriminatorColumn
public abstract class Item {
    @Id @GeneratedValue
    @Column(name = "ITEM_ID")
    private Long id;

    private String name;
    private int price;
}

@Entity
public class Album extends Item {
    private String artist;
}
```

저장하면 INSERT가 2번(ITEM, ALBUM), 조회하면 조인이 나간다.

```sql
insert into ITEM (NAME, PRICE, DTYPE, ITEM_ID) values (...)
insert into ALBUM (ARTIST, ITEM_ID) values (...)

select ... from ALBUM a inner join ITEM i on a.ITEM_ID = i.ITEM_ID where ...
```

| 장점 | 단점 |
|------|------|
| 테이블 정규화 | 조회 시 조인을 많이 사용, 성능 저하 |
| FK 참조 무결성 제약조건 활용 가능 | 조회 쿼리가 복잡함 |
| 저장공간 효율화 | 저장 시 INSERT SQL 2번 호출 |

---

## 3. 단일 테이블 전략 (SINGLE_TABLE) - 단순할 때의 선택지

자식 테이블을 만들지 않고 **모든 컬럼을 부모 테이블 하나에 통합**한다. `DTYPE`으로 타입을 구분한다.

```
ITEM (ITEM_ID PK, NAME, PRICE, ARTIST, DIRECTOR, ACTOR, AUTHOR, ISBN, DTYPE)
```

```java
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
public abstract class Item { ... }
```

전략만 바꾸면 되고 자식 코드는 그대로다. INSERT 1번, 조회도 조인 없이 1번.

| 장점 | 단점 |
|------|------|
| 조인이 필요 없어 일반적으로 조회 성능이 빠름 | 자식 엔티티가 매핑한 컬럼은 **모두 null 허용**이 됨 |
| 조회 쿼리가 단순함 | 테이블 하나가 계속 커져서 상황에 따라 조회가 오히려 느려질 수 있음 |

⚠️ `Album`을 저장하면 `AUTHOR`, `ISBN`, `DIRECTOR` 등 다른 자식 컬럼은 전부 null이 들어간다. not null 제약을 걸 수 없다는 게 구조적인 약점이다.

## 4. 구현 클래스마다 테이블 전략 (TABLE_PER_CLASS) - 쓰지 말 것

부모 테이블 없이 **자식 테이블마다 부모 컬럼을 전부 복사**해서 만든다.

```
ALBUM (ITEM_ID PK, NAME, PRICE, ARTIST)
MOVIE (ITEM_ID PK, NAME, PRICE, DIRECTOR, ACTOR)
BOOK  (ITEM_ID PK, NAME, PRICE, AUTHOR, ISBN)
```

> **이 전략은 데이터베이스 설계자와 ORM 전문가 둘 다 추천하지 않는다.**

치명적인 문제: 부모 타입으로 조회할 때다. `em.find(Item.class, id)`를 하면 어느 테이블에 있는지 모르므로 **모든 자식 테이블을 UNION으로 뒤진다.**

```sql
select ... from (
    select ..., 'ALBUM' as clazz from ALBUM
    union all select ..., 'MOVIE' as clazz from MOVIE
    union all select ..., 'BOOK' as clazz from BOOK
) where ITEM_ID = ?
```

| 장점 | 단점 |
|------|------|
| 서브타입을 명확하게 구분해서 처리할 때 효과적 | 여러 자식 테이블을 함께 조회하면 UNION SQL로 느림 |
| not null 제약조건 사용 가능 | 자식 테이블을 통합해서 쿼리하기 어려움 |

## 5. 전략 선택 기준

| 기준 | 선택 |
|------|------|
| 기본 | **조인 전략** — 정규화도 되고 설계가 깔끔함 |
| 테이블이 아주 단순하고 확장 가능성이 낮음 | 단일 테이블 전략 — 성능/단순함 우선 |
| 어떤 경우든 | 구현 클래스마다 테이블 전략은 배제 |

조인 전략과 단일 테이블 전략의 트레이드오프(정규화 vs 조회 단순함)를 이해하고 상황에 맞게 고르면 된다.

## 6. @MappedSuperclass - 공통 매핑 정보 모으기

상속관계 매핑과는 **전혀 다른 용도**다. 테이블과 매핑되지 않고, **자식 클래스에 매핑 정보만 물려주는** 클래스를 만들 때 쓴다.

전형적인 예: 모든 엔티티에 등록일/수정일/등록자/수정자를 넣고 싶을 때.

```java
@MappedSuperclass
public abstract class BaseEntity {
    private String createdBy;
    private LocalDateTime createdDate;
    private String lastModifiedBy;
    private LocalDateTime lastModifiedDate;
}

@Entity
public class Member extends BaseEntity { ... }  // MEMBER 테이블에 4개 컬럼이 추가된다
```

특징을 정확히 구분해야 한다.

| 항목 | @MappedSuperclass |
|------|-------------------|
| 상속관계 매핑인가 | ❌ 아니다 |
| 엔티티인가 | ❌ 아니다. 테이블과 매핑되지 않는다 |
| 역할 | 자식 클래스에 **매핑 정보만 제공** |
| 조회 가능한가 | ❌ `em.find(BaseEntity.class, ...)` 불가 |
| 권장 형태 | 직접 생성할 일이 없으므로 **추상 클래스** |

💡 참고: `@Entity` 클래스는 같은 `@Entity`이거나 `@MappedSuperclass`로 지정한 클래스만 상속할 수 있다. 그래서 상속관계 매핑(부모도 @Entity)과 @MappedSuperclass(부모는 매핑 정보만)가 문법상 둘 다 가능하고, 목적에 따라 골라 쓰는 것이다.

| 구분 | 상속관계 매핑 (@Inheritance) | @MappedSuperclass |
|------|------------------------------|-------------------|
| 목적 | 도메인의 진짜 상속 구조를 테이블로 표현 | 공통 필드 코드 중복 제거 |
| 부모가 테이블/엔티티인가 | O (JOINED면 테이블도 생김) | X |
| 부모 타입으로 조회 | O (`em.find(Item.class, ...)`) | X |

## 7. 실전 예제 4 - jpashop에 상속관계 적용

추가된 요구사항:

- 상품의 종류는 **음반, 도서, 영화**가 있고 이후 더 확장될 수 있다.
- 모든 데이터는 **등록일과 수정일**이 필수다.

이를 반영하면 도메인 모델에서 `Item`이 추상 부모가 되고, `Album`/`Book`/`Movie`가 자식이 된다. 테이블은 단일 테이블 전략 기준으로 `ITEM` 하나에 `DTYPE` + 자식 컬럼들이 합쳐진다.

내가 실제로 적용한 코드다. 6장의 `categoryItems` 연관관계를 유지한 채 추상 클래스로 승격했다.

```java
// Item.java — 요구사항 1: 상속관계 매핑 (단일 테이블 전략)
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "DTYPE") // 생략 가능, 자동 생성
public abstract class Item extends BaseEntity {
    @Id @GeneratedValue
    @Column(name = "ITEM_ID")
    private Long id;

    private String name;
    private int price;
    private int stockQuantity;

    @OneToMany(mappedBy = "item")
    private List<CategoryItem> categoryItems = new ArrayList<>();
}

@Entity
public class Album extends Item {
    private String artist;
    private String etc;
}

@Entity
public class Book extends Item {
    private String author;
    private String isbn;
}

@Entity
public class Movie extends Item {
    private String director;
    private String actor;
}
```

```java
// BaseEntity.java — 요구사항 2: 등록일/수정일을 @MappedSuperclass로 전 엔티티에 공통 적용
@MappedSuperclass
public class BaseEntity {
    private String createdBy;
    private LocalDateTime createdDate;
    private String lastModifiedBy;
    private LocalDateTime lastModifiedDate;
}
```

`Member`, `Order`, `OrderItem`, `Delivery`, `Category` 전부 `extends BaseEntity`만 붙여서 각 테이블에 등록/수정 컬럼 4개가 생기는 것을 확인했다.

실제로 Book을 저장하면 ITEM 테이블 한 행에 이렇게 들어간다.

```java
Book book = new Book();
book.setName("JPA");
book.setAuthor("김영한");
em.persist(book);
// ITEM: DTYPE=Book, NAME=JPA, AUTHOR=김영한, ARTIST=null, DIRECTOR=null, ...
```

⚠️ 실습에서 확인한 주의점들:

- **다른 자식의 컬럼은 전부 null로 들어간다** — 단일 테이블 전략의 특성. 그래서 자식 필드는 not null 제약을 걸 수 없다. 단, 부모 `Item`의 `price` 같은 **프리미티브(`int`) 필드는 DDL에 not null이 붙고**, 값을 안 넣으면 기본값 0이 저장된다. null이 필요한 필드는 `Integer` 같은 래퍼 타입을 써야 한다.
- 6장까지의 `Item`은 `OrderItem`, `CategoryItem`이 `@ManyToOne`으로 참조하고 있는데, `Item`을 추상 클래스로 바꿔도 이 참조들은 수정할 필요가 없다 — 다형적 참조가 그대로 동작하는 것이 상속관계 매핑의 장점이다.
- 강의 권장은 `BaseEntity`를 **추상 클래스**로 만드는 것이다 (직접 생성할 일이 없으므로). 내 코드는 일반 클래스로 뒀는데, `@Entity`가 아니라서 단독 저장은 어차피 불가능하지만 `new BaseEntity()` 자체를 막으려면 `abstract`가 낫다.

## ✅ 핵심 요약

1. RDB에는 상속이 없다. 슈퍼타입-서브타입 모델을 3가지 전략으로 매핑한다.
2. 전략은 부모의 `@Inheritance(strategy = ...)` 한 줄로 바뀌고, 객체 코드는 그대로다.
3. **조인 전략이 기본.** 정규화, FK 무결성, 저장 효율이 장점이고 조인 비용이 단점.
4. **단일 테이블 전략은 단순할 때.** 조회가 빠르지만 자식 컬럼이 전부 null 허용이 된다.
5. **구현 클래스마다 테이블 전략은 쓰지 않는다.** 부모 타입 조회 시 UNION이 나간다.
6. `@MappedSuperclass`는 상속관계 매핑이 아니라 **공통 매핑 정보 제공용**이다. 엔티티가 아니고, 조회도 불가하며, 추상 클래스로 만든다. 등록일/수정일 같은 공통 필드에 쓴다.
