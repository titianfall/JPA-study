## 1. JPA의 데이터 타입 분류

JPA는 데이터 타입을 최상위 레벨에서 딱 두 가지로 나눈다.

| 분류 | 설명 | 예 |
|------|------|-----|
| **엔티티 타입** | `@Entity`로 정의하는 객체. **식별자(id)가 있어서** 데이터가 변해도 추적 가능 | 회원 엔티티의 키나 나이 값을 변경해도 id로 인식 가능 |
| **값 타입** | `int`, `Integer`, `String`처럼 단순히 값으로 사용하는 자바 기본 타입이나 객체. **식별자가 없어** 변경 시 추적 불가 | 숫자 100을 200으로 변경하면 완전히 다른 값으로 대체됨 |

### 값 타입의 3분류

1. **기본값 타입** — 자바 기본 타입(`int`, `double`), 래퍼 클래스(`Integer`, `Long`), `String`
2. **임베디드 타입** (embedded type, 복합 값 타입)
3. **컬렉션 값 타입** (collection value type)

### 기본값 타입의 특징

- 생명주기를 **엔티티에 의존**한다. (예: 회원을 삭제하면 이름, 나이 필드도 함께 삭제)
- 값 타입은 **공유하면 안 된다**. (예: 회원 이름 변경 시 다른 회원의 이름도 함께 변경되면 안 됨)

> 💡 **자바의 기본 타입은 절대 공유되지 않는다.**
> `int`, `double` 같은 primitive type은 대입 시 항상 **값을 복사**한다.
> `Integer` 같은 래퍼 클래스나 `String`은 참조가 공유될 수는 있지만 **변경할 방법이 없다**(불변). 그래서 안전하다.

## 2. 임베디드 타입 (복합 값 타입)

새로운 값 타입을 **직접 정의**할 수 있다. 주로 기본값 타입들을 모아서 만들기 때문에 **복합 값 타입**이라고도 부른다. 임베디드 타입도 결국 `int`, `String`과 같은 **값 타입**이다 (엔티티 아님!).

### 왜 쓰나?

```
[임베디드 타입 적용 전]              [임베디드 타입 적용 후]
Member                              Member
 ├ id                                ├ id
 ├ name                              ├ name
 ├ startDate      ─┐                 ├ workPeriod  (Period)
 ├ endDate        ─┘ 근무 기간        └ homeAddress (Address)
 ├ city           ─┐
 ├ street          │ 집 주소
 └ zipcode        ─┘
```

"회원은 **이름, 근무 기간, 집 주소**를 가진다"처럼 **추상화된 설명이 가능**해진다.

### 사용법 — 어노테이션 2개 (커밋 `d1024a3`)

| 어노테이션 | 붙이는 곳 |
|-----------|----------|
| `@Embeddable` | 값 타입을 **정의**하는 클래스 |
| `@Embedded` | 값 타입을 **사용**하는 엔티티의 필드 |

⚠️ 임베디드 타입 클래스는 **기본 생성자가 필수**다. (Hibernate가 리플렉션으로 객체를 만들 때 사용 — 직접 호출을 막고 싶으면 `protected`로)

내 코드 (`hellojpa/Period.java`):

```java
@Embeddable
public class Period {
    private LocalDateTime startDate;
    private LocalDateTime endDate;

    public Period() {}

    // 입사 시점엔 endDate를 모를 수 있으니 startDate만 받는 생성자도 제공
    public Period(LocalDateTime startDate) {
        this.startDate = startDate;
    }

    // 값 타입 안에 의미 있는 비즈니스 메서드를 만들 수 있다
    public boolean isWork() {
        LocalDateTime now = LocalDateTime.now();
        return startDate.isBefore(now) && endDate.isAfter(now);
    }
}
```

내 코드 (`hellojpa/Member.java`):

```java
@Entity
public class Member {
    @Id @GeneratedValue(strategy = GenerationType.AUTO)
    @Column(name = "MEMBER_ID")
    private Long id;

    // 기간 Period
    @Embedded
    private Period workPeriod;

    // 주소
    @Embedded
    private Address homeAddress;
}
```

> ⚠️ 강의(구버전)는 `javax.persistence`지만 내 프로젝트는 `jakarta.persistence` 사용.
> `LocalDateTime`은 Hibernate 6에서 `@Temporal` 없이 TIMESTAMP로 자동 매핑된다.

### 임베디드 타입의 장점

- **재사용** — Address를 여러 엔티티(Member, Delivery)에서 사용 가능
- **높은 응집도** — 관련 필드가 한 클래스에 모임
- `Period.isWork()`처럼 **해당 값 타입만 사용하는 의미 있는 메서드**를 만들 수 있음 (객체지향적)
- 임베디드 타입을 포함한 모든 값 타입은, 값 타입을 소유한 **엔티티에 생명주기를 의존**함

### 임베디드 타입과 테이블 매핑

- 임베디드 타입은 **엔티티의 값일 뿐**이다. → MEMBER 테이블에 `city`, `street`, `zipcode`, `startDate`, `endDate` **컬럼으로 들어간다** (별도 테이블 X).
- 임베디드 타입을 사용하기 **전과 후에 매핑하는 테이블은 같다.**
- 객체와 테이블을 아주 세밀하게(fine-grained) 매핑하는 것이 가능해진다.
- 잘 설계한 ORM 애플리케이션은 매핑한 테이블 수보다 **클래스 수가 더 많다.**

### 임베디드 타입과 연관관계

임베디드 타입은 내부에 다른 임베디드 타입이나 **엔티티를 참조**할 수도 있다.

> ⚠️ **내 환경(Hibernate 6.x)과 강의의 차이**: 임베디드 타입 안에 엔티티 필드를 매핑 어노테이션 없이 그냥 두면
> `Could not determine recommended JdbcType` 에러가 난다. 구버전은 느슨하게 넘어갔지만
> Hibernate 6부터 타입 해석이 엄격해졌다. 참조하려면 `@ManyToOne` + `@JoinColumn`을 명시하거나,
> 매핑 대상이 아니면 `@Transient`를 붙여야 한다.

### `@AttributeOverride` — 속성 재정의

한 엔티티에서 **같은 값 타입을 두 번** 사용하면 컬럼명이 중복돼서 매핑 오류가 발생한다.

```java
@Embedded
@AttributeOverrides({
        @AttributeOverride(name = "city",    column = @Column(name = "WORK_CITY")),
        @AttributeOverride(name = "street",  column = @Column(name = "WORK_STREET")),
        @AttributeOverride(name = "zipcode", column = @Column(name = "WORK_ZIPCODE"))
})
private Address workAddress; // 컬럼명을 재정의해서 중복 회피
```

> ⚠️ 실습 중 겪은 컴파일 에러 2가지:
> - `@Column("WORK_CITY")` → **에러**. `@Column`은 기본 속성(`value`)이 없어서 반드시 `name = "..."` 형태로 써야 한다.
> - `@AttributeOverride`의 `name`은 **값 타입 클래스의 필드명**과 정확히 일치해야 한다.

### 임베디드 타입과 null

임베디드 타입의 값이 `null`이면 매핑한 **컬럼 값은 모두 null**이 된다.

## 3. 값 타입과 불변 객체 (커밋 `5e819e7`, `e002a56`)

> 값 타입은 복잡한 객체 세상을 조금이라도 단순화하려고 만든 개념이다.
> 따라서 값 타입은 **단순하고 안전하게** 다룰 수 있어야 한다.

### ⚠️ 값 타입 공유 참조 — 부작용(side effect)의 위험

내 실습 (`JpaMain.java`) — 동일한 주소 인스턴스를 두 회원이 공유:

```java
Address address = new Address("city", "street", "10000"); // 힙에 인스턴스 딱 1개

member1.setHomeAddress(address); // member1 → 그 인스턴스를 가리킴
member2.setHomeAddress(address); // member2 → "같은" 인스턴스를 가리킴

em.persist(member1);
em.persist(member2);

// member1의 주소만 바꾸고 싶었는데...
member1.getHomeAddress().setCity("newCity");
// → 같은 객체라 member2의 주소도 함께 바뀜 → UPDATE 쿼리가 "2번" 나간다!
```

**섞이는 주체는 영속성 컨텍스트가 아니라 자바 객체 참조다.** `em.persist()`는 Address를 복사해서 따로 보관하지 않는다 — 임베디드 타입은 엔티티의 값(컬럼 몇 개)일 뿐이라, 두 회원은 여전히 힙의 동일한 Address 하나를 공유한다. flush 시점에 더티 체킹이 "둘 다 변경됨"으로 판단해 UPDATE 2방이 나가는 것. 이런 부작용 버그는 **컴파일 시점에 잡을 수 없어서** 찾기가 정말 어렵다.

### 객체 타입의 한계 — 참조 전달

```java
// 기본 타입 — 값을 복사
int a = 10;
int b = a;   // b는 복사된 값
b = 4;       // a는 그대로 10

// 객체 타입 — 참조를 전달 (== 비교는 "같은 인스턴스를 가리키는가"만 본다)
Address a = new Address("Old");
Address b = a;        // 같은 인스턴스를 가리킴
b.setCity("New");     // a도 같이 바뀜!
```

값을 복사(`new Address(a.getCity(), ...)`)해서 쓰면 피할 수 있지만, **객체의 공유 참조 자체는 컴파일러가 막아주지 않는다.**

### 해결 — 불변 객체 (근본적 해결)

- **값 타입은 불변 객체(immutable object)로 설계해야 한다.**
- 불변 객체: **생성 시점 이후 절대 값을 변경할 수 없는 객체**
- 만드는 법: **생성자로만 값을 설정하고 수정자(Setter)를 만들지 않으면 됨**

내 코드 (`hellojpa/Address.java`) — Setter를 `private`으로 내려서 외부 수정 차단:

```java
@Embeddable
public class Address {
    private String city;
    private String street;
    private String zipcode;

    public Address() {}

    public Address(String city, String street, String zipcode) {
        this.city = city;
        this.street = street;
        this.zipcode = zipcode;
    }

    public String getCity() { return city; }
    // ...

    private void setCity(String city) { this.city = city; } // 외부 호출 불가
    // (필드 접근 방식에서는 private setter도 사실 없어도 됨)
}
```

값을 바꾸고 싶으면? → **새 객체로 통째로 교체** (내 실습 코드):

```java
// 값 타입의 수정(update) — 유지할 값은 기존 객체에서 꺼내온다
Address address = findMember.getHomeAddress();
findMember.setHomeAddress(new Address("newCity", address.getStreet(), address.getZipcode()));
```

임베디드 타입은 주인 테이블의 컬럼이므로 이 교체는 **평범한 UPDATE 1번**으로 반영된다.

> 💡 참고: `Integer`, `String`은 자바가 제공하는 대표적인 불변 객체다.
>
> **"불변이라는 작은 제약으로 부작용이라는 큰 재앙을 막을 수 있다."**

## 4. 값 타입의 비교 (커밋 `8d68be2`)

값 타입은 인스턴스가 달라도 그 안의 값이 같으면 **같은 것으로 봐야 한다.**

| 비교 방식 | 의미 | 사용 |
|-----------|------|------|
| **동일성(identity) 비교** | 인스턴스의 **참조 값**(같은 객체를 가리키는가)을 비교 | `==` |
| **동등성(equivalence) 비교** | 인스턴스의 **값**을 비교 | `equals()` |

```java
Address a = new Address("서울시");
Address b = new Address("서울시");
// a == b        → false (서로 다른 인스턴스)
// a.equals(b)   → equals()를 재정의해야 true
```

⚠️ `Object.equals()`의 기본 구현은 `this == obj`라서, **재정의하지 않으면 `equals()`도 `==`와 똑같이 동작한다.**

내 코드 (`jpashop/domain/Address.java`) — 모든 필드 기준 재정의:

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (o == null || getClass() != o.getClass()) return false;
    Address address = (Address) o;
    // getter를 통해 equals를 구현해야 함. 프록시일때는 (필드 직접 접근으로) 계산이 안되기 때문임
    return Objects.equals(city, address.city) &&
            Objects.equals(street, address.street) &&
            Objects.equals(zipcode, address.zipcode);
}

@Override
public int hashCode() {
    return Objects.hash(getCity(), getStreet(), getZipcode());
}
```

> 💡 `equals()`를 재정의하면 `hashCode()`도 함께 재정의해야 `HashMap`/`HashSet`에서 정상 동작한다.
> IntelliJ `Alt + Insert` → *equals() and hashCode()* 자동 생성 활용.

## 5. 값 타입 컬렉션 (커밋 `8aee8f8`, `d4cc3d1`)

값 타입을 **하나 이상(컬렉션에) 저장**할 때 사용한다. 관계형 DB는 테이블 안에 컬렉션을 저장할 수 없으므로 **별도 테이블**로 풀어낸다.

내 코드 (`hellojpa/Member.java`):

```java
// ElementCollection 의 fetch default는 FetchType.LAZY 이다.
// 선호 음식
@ElementCollection(fetch = FetchType.LAZY)
@CollectionTable(name = "FAVORITE_FOOD", joinColumns = {
        @JoinColumn(name = "MEMBER_ID") // 외래키로 인식
})
@Column(name = "FOOD_NAME") // 예외적으로 가능 - column 이 하나이기 때문
private Set<String> favoriteFoods = new HashSet<>();

// 집 주소들 - 값 타입 매핑
@ElementCollection(fetch = FetchType.LAZY)
@CollectionTable(name = "ADDRESS", joinColumns = {
        @JoinColumn(name = "MEMBER_ID")
})
private List<Address> addressesHistory = new ArrayList<>();
```

- `Set`을 쓴 이유: 선호 음식은 중복/순서가 의미 없음. `List`를 쓴 이유: 주소 "이력"은 순서가 의미 있음.
- `Set`은 인터페이스라 `new HashSet<>()` 구현체로 초기화. **컬렉션 필드는 선언과 동시에 초기화**하는 게 관례 (안 하면 `new Member()` 직후 `getXxx().add(...)`에서 NPE — 실습 중 직접 겪음).

```
MEMBER                    FAVORITE_FOOD              ADDRESS
┌───────────┐             ┌──────────────┐          ┌──────────────┐
│ MEMBER_ID │◀────────────│ MEMBER_ID(FK)│          │ MEMBER_ID(FK)│
│ city 등   │             │ FOOD_NAME    │          │ CITY         │
└───────────┘             └──────────────┘          │ STREET       │
 (임베디드는 여기 컬럼)     (PK: 모든 컬럼)            │ ZIPCODE      │
                                                    └──────────────┘
```

### 조회 — 지연 로딩 확인 (커밋 `8aee8f8`)

`em.find(Member.class, id)` 시 SELECT는 **MEMBER 테이블 1방만** 나간다 (임베디드 타입은 같은 행이라 함께 조회됨). 값 타입 컬렉션은 **기본이 지연 로딩** — 컬렉션을 실제로 사용하는 순간 각자 SELECT가 나간다.

```java
Member findMember = em.find(Member.class, member.getId()); // SELECT Member (1방)
System.out.println("=================== find End ============");
findMember.getAddressesHistory().remove(...);              // 이 순간 SELECT ADDRESS
findMember.getFavoriteFoods().remove("치킨");               // 이 순간 SELECT FAVORITE_FOOD
```

### 수정 — 실제 SQL 로그로 확인한 동작 (커밋 `d4cc3d1`)

내 실습 코드:

```java
// 치킨 > 한식 : delete(단일) > insert(신규)
findMember.getFavoriteFoods().remove("치킨");
findMember.getFavoriteFoods().add("한식");

// 주소 수정 : delete(전체) > insert(기존) > insert(신규)
findMember.getAddressesHistory().remove(new Address("old1", "street1", "20000"));
findMember.getAddressesHistory().add(new Address("newCity1", "street4", "40000"));
```

실제로 나간 SQL:

```sql
-- Set<String> : 모든 컬럼이 PK라 행 특정 가능 → 그 행만 콕 집어 처리
DELETE FROM FAVORITE_FOOD WHERE MEMBER_ID=? AND FOOD_NAME=?  -- 치킨만 삭제
INSERT INTO FAVORITE_FOOD (MEMBER_ID, FOOD_NAME) VALUES (?, ?) -- 한식만 추가

-- List<Address> : 행 특정 불가 → 통째로 재생성(recreate)
/* one-shot delete */ DELETE FROM ADDRESS WHERE MEMBER_ID=?   -- 전체 삭제
INSERT INTO ADDRESS (MEMBER_ID, city, street, zipcode) VALUES (?, ?, ?, ?) -- old2 (기존)
INSERT INTO ADDRESS (MEMBER_ID, city, street, zipcode) VALUES (?, ?, ?, ?) -- newCity1 (신규)
```

- `List<Address>`는 변경이 생기면 **전체 DELETE → commit 시점 컬렉션 내용 전부 재INSERT**. INSERT 개수 = 최종 원소 개수 (기존/신규 구분 없음). 개념적으로 `clear()` 후 전부 다시 쓰는 것과 같은 SQL이다.
- Hibernate는 **diff를 낼 수 있으면 diff만큼만, 못 내면 재생성으로 후퇴**한다. `List` + 임베디드는 순서/중복 허용 bag이라 DB 행과 메모리 원소를 1:1 대응시킬 수 없다 → 근본 원인은 역시 **값 타입에는 식별자가 없다**는 것.

### ⚠️ equals의 "조용한 실패" (커밋 `d4cc3d1` — 실습에서 직접 겪은 것)

컬렉션 `remove(Object)`는 **`equals()`로 대상을 찾는다.** 실습 중 두 번 데였다:

1. **equals 재정의 전**: `remove(new Address(...))`가 참조 비교로 동작 → 아무것도 안 지워짐 → 컬렉션이 스냅샷과 차이가 없으니 **SQL이 아예 안 나감**. (변경 감지는 "메서드 호출"이 아니라 "스냅샷과의 차이"에서 SQL을 만든다)
2. **equals 재정의 후에도**: 저장한 값은 `("old1", "street1", "20000")`인데 `("old1", "street", "10000")`으로 지우려 함 → 한 필드라도 다르면 매칭 실패 → remove 실패했는데 add는 성공해서 **INSERT가 3개** 나가는 걸 보고서야 발견.

> 💡 예외도 로그도 없이 실패하므로 **`remove()`의 boolean 반환값을 확인**하는 습관을 들이자.
> ```java
> boolean removed = list.remove(new Address("old1", "street1", "20000"));
> System.out.println("removed = " + removed); // false면 equals 미구현 or 값 불일치
> ```
> 또한 `remove(List.of("치킨"))`처럼 원소가 아닌 컬렉션을 넘겨도 타입 불일치로 조용히 실패한다.
> 리스트로 여러 개 지우려면 `removeAll(List.of(...))`.

### ⚠️ 값 타입 컬렉션의 제약사항 정리

- 값 타입은 식별자 개념이 없다 → 값을 변경하면 **추적이 어렵다.**
- 변경 발생 시 주인 엔티티와 연관된 **모든 데이터를 삭제하고 현재 값을 모두 다시 저장**한다 (`List` + 임베디드 기준).
- 값 타입 컬렉션 테이블은 **모든 컬럼을 묶어 PK 구성** → null 입력 X, 중복 저장 X.
- 값 타입 컬렉션은 영속성 전이(Cascade) + 고아 객체 제거를 **별도 설정 없이 기본으로** 가진다.

### 값 타입 컬렉션의 대안 — 엔티티로 승격 (커밋 `f47222c`)

실무에서는 상황에 따라 값 타입 컬렉션 대신 **일대다 관계 + 엔티티**를 고려한다.

내 코드 (`hellojpa/AddressEntity.java`):

```java
@Entity
@Table(name = "ADDRESS")
public class AddressEntity {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;          // 식별자가 생김 → 행 특정 가능, 변경 추적 가능

    private Address address;  // 진짜 값 타입은 안에서 사용

    public AddressEntity(String city, String street, String zipcode) {
        this.address = new Address(city, street, zipcode);
    }
}
```

내 코드 (`hellojpa/Member.java`):

```java
// 집 주소들 - entity 매핑
@OneToMany(fetch = FetchType.LAZY, cascade = CascadeType.ALL, orphanRemoval = true)
@JoinColumn(name = "MEMBER_ID")
private List<AddressEntity> addressHistory = new ArrayList<>();
```

- **영속성 전이(Cascade) + 고아 객체 제거**로 값 타입 컬렉션처럼 쓴다 (8장에서 배운 것의 조합).
- `AddressEntity`는 `Address`와 남남인 별개 타입이므로 컬렉션에 넣을 때 **감싸서** 넣는다:
  `member.getAddressHistory().add(new AddressEntity("old1", "street2", "20000"));`
- id로 행을 특정할 수 있으므로 컬렉션에서 제거하면 전체 재INSERT 대신 **그 행만 DELETE** 된다.

> 💡 값 타입 컬렉션은 정말 단순한 경우에만: select box에서 치킨/피자 다중 선택처럼
> **추적할 필요 없고, 값이 바뀌어도 업데이트할 필요 없는** 수준.
> 실무 수정 패턴도 diff 계산 없이 `clear()` + 전체 재등록이 흔하다 —
> 개별 remove를 정교하게 써야 하는 상황 자체가 "엔티티로 승격할 신호"다.

## 6. 정리 — 엔티티 타입 vs 값 타입

| | 엔티티 타입 | 값 타입 |
|---|------------|---------|
| 식별자 | **O** | **X** |
| 생명 주기 | 스스로 관리 | **엔티티에 의존** |
| 공유 | 가능 | **공유하지 않는 것이 안전** (복사해서 사용) |
| 변경 | 가능 | **불변 객체로 만드는 것이 안전** |

> ⚠️ **값 타입은 정말 값 타입이라 판단될 때만 사용하라.**
> - 엔티티와 값 타입을 혼동해서 **엔티티를 값 타입으로 만들면 안 된다.**
> - 식별자가 필요하고, 지속해서 값을 추적·변경해야 한다면 그것은 값 타입이 아닌 **엔티티**다.

## 7. 실전 예제 6 — 값 타입 매핑 (jpashop, 커밋 `9344246`)

주문 도메인에서 `Member`와 `Delivery`가 각각 갖고 있던 `city`, `street`, `zipcode` 필드를 `Address` 값 타입으로 추출했다.

내 코드 (`jpashop/domain/Address.java`):

```java
@Embeddable
public class Address {
    @Column(length = 10)   // 값 타입에도 @Column 등 매핑 정보 사용 가능 — 길이 제약을 한 곳에서 관리
    private String city;
    @Column(length = 20)
    private String street;
    @Column(length = 5)
    private String zipcode;

    // 원하는 기능의 비즈니스 메서드 생성 가능
    private String fullAddress() {
        return getCity() + getStreet() + getZipcode();
    }

    // 기본 생성자 + 전체 생성자, Getter만 public (Setter는 private → 불변)
    // equals()/hashCode() 재정의 (4. 참고)
}
```

내 코드 (`jpashop/domain/Member.java`, `Delivery.java`):

```java
@Entity
public class Member extends BaseEntity {
    // 기존: private String city; private String street; private String zipcode;
    @Embedded
    private Address address;
}

@Entity
public class Delivery extends BaseEntity {
    @Embedded
    private Address deliveryAddress; // Member와 재사용!
}
```

- `Address`를 값 타입으로 추출하니 **Member와 Delivery에서 재사용**되고, 주소 검증/길이 제약/유틸 메서드가 한 곳에 모였다.
- 테이블 구조는 그대로 (MEMBER, DELIVERY에 각각 컬럼으로 들어감).

## ✅ 핵심 요약

1. JPA 데이터 타입은 **엔티티 타입**(식별자 O, 추적 가능)과 **값 타입**(식별자 X, 엔티티에 생명주기 의존)으로 나뉜다.
2. 값 타입은 **기본값 타입 / 임베디드 타입 / 컬렉션 값 타입** 3가지.
3. 임베디드 타입은 `@Embeddable`(정의) + `@Embedded`(사용), **기본 생성자 필수**, 별도 테이블 없이 주인 테이블의 컬럼으로 매핑.
4. 같은 값 타입을 한 엔티티에서 두 번 쓰면 `@AttributeOverrides`로 컬럼명 재정의 (`@Column`은 `name =` 필수).
5. 값 타입 공유는 **부작용(side effect)** 을 낳는다 — 섞이는 건 영속성 컨텍스트가 아니라 **자바 참조**. 근본 해결은 **불변 객체**(생성자만, Setter 금지). 수정은 새 객체로 교체(임베디드는 UPDATE 1번).
6. 값 타입 비교는 `==`(동일성)이 아니라 **`equals()`(동등성)** — `equals()`/`hashCode()` 재정의 필수. 컬렉션 `remove()`도 equals 기반이라 **미구현/값 불일치 시 조용히 실패**한다 (반환값 확인!).
7. 값 타입 컬렉션(`@ElementCollection`)은 기본 지연 로딩이며, `List`+임베디드는 변경 시 **전체 삭제 후 재INSERT** (`Set`+기본값은 단일 행 처리로 최적화되기도 함).
8. 추적/변경이 필요한 데이터는 **일대다 엔티티(id 보유) + Cascade + orphanRemoval**로 승격 — **식별자가 필요하면 그것은 엔티티다.**
